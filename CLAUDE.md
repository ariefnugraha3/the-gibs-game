## Overview

Browser **TOP-DOWN SHOOTER** (Alien Shooter-style; **pivoted from FPS on 2026-07-11** — see the "camera is a logic pivot" rule below): "Gibran vs Robot 3D", Three.js r128. **Controls: WASD = screen-axis movement, mouse = virtual aim cursor (bullets follow it), LMB = shoot/use, RMB = move-to-cursor-point (WASD cancels), 1/2/3 = the three weapon slots, Q cycles weapons, 4 medkit, F melee, Shift dodge/evade (brief-invincible tumble). Crouch, jump, ADS (2026-07-11 pivot), sprint (replaced by the dodge 2026-07-11), R/reload (magazine system removed 2026-07-11) and the thrown grenade (replaced by the Grenade Launcher weapon 2026-07-11) do not exist.** **All user-facing UI text MUST be in English — permanent user rule (2026-07-05), no exceptions, applies to every future change.** Code comments are in Indonesian. **Two game modes** picked on the `#modeSelect` screen at load (plus an EASY/NORMAL/HARD difficulty row — `applyDifficulty` recomputes CFG from the pristine `CFG_BASE`, per-difficulty high scores): **Survival** (round-based waves — you start with **only a pistol (full ammo pool)**, wave 1 spawns 15 robots [`robotsPerWaveBase` + `Step`·(n−1)], clearing a wave shows "WAVE CLEARED" + a `shopCountdownSec` countdown, then the between-wave **Field Shop** auto-opens [a clickable mouse-driven menu — buy ammo/health refills, a **Medkit** (held, up to 2; equip with 4, then hold left-click 2 s to use for +70% HP — slowed & no sprint while using), Heal/Strengthen Monas, the **Radar** (Survival starts without it), and the Shotgun, Assault Rifle & **Grenade Launcher** (2000 pts) — **max 3 weapons, buying a 4th prompts which to replace**; click an item then its **Buy** button, then **Start Next Wave** → "Are you ready?" confirm]; **robots prioritize the Monument** — every robot attacks Monas by default and only aggros onto the player when the player is within `playerAggroMeters` (15 m) of that robot (returning to Monas past it) — and once a robot reaches Monas and starts hitting it, its aggro SHRINKS to `monasLockAggroMeters` (5 m) and a `monasLockChance` (50%) roll may LOCK it fully onto Monas (never chases the player again), `monasHp` 0 = "THE MONUMENT HAS FALLEN"; score = shop currency; from wave 4 on, each wave has a rising chance to trigger a thick grey Monas-fog event) and **Campaign**, a linear 3-stage mode — Stage 1 "Gedung Terbengkalai (Lantai 2)" (indoor 60×60 m grid building; ## Architecture (refactored 2026-07-05 into ES modules)

The old single-file `index.html` was split into **ES modules under `src/`** with a **scene system**. There is still **no build step, no npm dependencies, no framework** — plain static files; Three.js r128 + post-processing examples come from CDN `<script>` tags as the global `THREE` (modules do NOT import it). `package.json` is metadata-only (`"type": "module"` for Node tooling).

**→ The complete module catalog, every export, the scene interface contract, the config key table, and the how-to-add-a-stage recipe live in [MODULES.md](MODULES.md). Read it before hunting through files.**

Load-bearing rules (details in MODULES.md):

- **Scenes:** `menu` (DOM-only) → `survivalScene` or `campaign stage1Scene → stage2Scene`, managed by `core/sceneManager.js` (`setScene`/`activeScene`). All mode-specific behavior (collision walls, ground height, bullet blockers, grenade bounds, robot movement AI, radar landmarks, HUD status, drop clamping, win check, restart policy) is behind `activeScene.*` hooks — **shared systems contain zero mode if-else**. Adding a campaign stage = one new file in `src/scenes/campaign/` + wiring per the MODULES.md recipe.
- **Both campaign stage worlds coexist in one THREE.Scene** ~26 km apart (building at x≈30000); `camera.far` 4000 + robot distance culling hide the far one. Campaign restart (death) always re-enters stage 1, which re-places both stages' robots + supplies (arrays are cleared by `resetGame` first). Every campaign robot carries `z.stage`; stage transition disposes the previous stage's robots silently.
- **All gameplay-tunable numbers live in `config/gameplay.json`**, loaded into `CFG` (`core/config.js`) before the game starts. Read `CFG.x.y` inside functions only (never at module top level). New tuning values go in the JSON, not as code constants. Visual-only values (FOV, animation amplitudes, colors) intentionally stay in code. (The reload keyframe auto-scaling note is historical — reload was removed with the magazine system 2026-07-11.)
- **Cross-module state = live ESM bindings**: the owning module exports `let` + setters (`isPaused`/`score` in `core/state.js`, `stamina`/`dodgeActive`/`dodgeProgress` in `entities/player.js`, `currentWeapon`/`isAiming`/`switchAnim` in `entities/weapons.js`, `scene`/`camera`/`renderer` in `core/renderer.js`). Circular imports are fine and used (weapons↔player, robots↔effects, stage1↔stage2, game↔input) **as long as bindings are only used inside functions**.
- **`updateGame()` block order in `core/game.js` is a contract** (matches the old `update()`): scene updateMode → weapon timers → player movement → weapon state → shooting → grenades → explosions → blood → drops → bullets → robots → win check. Bullets must move before the robot sweep hit-test.
- **The old `camera` is now a LOGIC PIVOT, not the renderer (top-down pivot 2026-07-11).** Its position = the player's eye-height point (feet + `eyeHCur`, same semantics as the FPS era) and its yaw = the AIM direction toward the cursor (set every frame by `updateTopdownAim` in input.js). Every gameplay system (robot targeting/claws, bullet/grenade/melee direction via `camera.getWorldDirection`, pickups, scene collision, radar, spawn points set by scenes) still reads `camera` and is UNCHANGED. Rendering is done by **`viewCam`** (renderer.js), a separate camera at fixed offset `CAM_OFF` (0, 116, 100 — brought closer in steps 2026-07-11 [185/100 → 161/87 → 146/79 → 130/82]; then 130/82 → 116/100 LOWERED the angle a further ~15% while KEEPING the distance ~154 units [y down, z up] so only the pitch flattens, not the zoom: ~57.8° → ~49.2°) — Alien Shooter oblique angle, fixed yaw facing −z, so **screen-up = world −z**. `followViewCam()` tracks a **dead-zone focus point** `camFocus` (not the pivot directly): it only shifts when the pivot leaves a tolerance box of world half-extents `DEAD_X`/`DEAD_Z` (`DEAD_Z` for north/south is intentionally smaller so the widescreen forward view isn't cramped), so small movement doesn't move the camera (anti-dizzy, per user); y follows without tolerance; jumps > 400 units (spawn/scene-change/restart) snap-recenter. The cursor raycast (`updateTopdownAim`) reads the actual viewCam matrix so aim stays correct with the offset. A visible **player avatar** (`entities/playerAvatar.js`, a **stylized low-poly soldier** — rounded cylinder/ellipsoid build matching the robots, overhauled 2026-07-11 from blocks) stands at the pivot's feet, faces `aimPoint`, walk-animates by real speed (`legL/legR` hip pivots), and carries the gun whose `avatarGunTip` (kept at its exact calibrated offset — bullet spawn point must not move) is the bullet spawn point + muzzle flash parent. The FPS viewmodel rigs are still BUILT (reload/switch/medkit logic reads their transforms) but live inside `fpsHolder`, a permanently invisible camera child — visible=false on the PARENT is immune to the `mesh.visible=true` writes of startSwitch/applyStartLoadout. Frame order in `animate()`: `updateTopdownAim` → `updateGame` → `followViewCam` + `updatePlayerAvatar` → decor → render. Decor animation (`world/decor.js` registry + weapon visuals) runs every frame **even while paused**.

## Key mechanics and conventions

- **Collisions are distance checks, not physics.** Robot-hit, pickup, and blast use `distanceTo` radii. **Bullet-vs-robot is a swept-segment check** (`segPointDist2`, prev→current position; spawn-frame segment starts at the player's EYE so point-blank never tunnels). Survival: Monas is a solid AABB resolved PER-AXIS (`resolveMonas` in world.js — `MONAS_HALF` 22, base 44×44; hug-and-slide, NOT a full revert, so the player never sticks — mirrors the campaign wall fix), solid trees/fountain cylinders (`resolveObstacles`), fence clamp. **`bulletBlocked` follows the real Monas silhouette height-by-height** (base 44 low → thin obelisk ~16 at eye height, nothing above y 64) so bullets pass through the empty air beside the obelisk and hit robots behind — do NOT revert it to a flat 44-wide column. Campaign: walkable-union tests (`stage1Walk` grid / `highwayWalk`) with per-axis `slideWalk` (hug walls, don't stick) + rotated-AABB `resolveBlockers` (medians/wrecks/furniture; standable tops are walkable). Robot bodies are solid to the player only (push-out at `CFG.robot.bodyBlockRadius` 7.5, deliberately < claw-stop 8.0 so an attacking robot doesn't shove the player every frame).
- **Frame-rate independence:** `dt` (clamped 0.05) and `step = dt*60`. Multiply motion by `step`; decrement timers by `dt`. Fire-rate/reload use real `Date.now()` time.
- **Entity model:** five arrays in `core/state.js` — `bullets, robots, grenades, explosions, drops` — iterated backwards for splicing. `explosions` doubles as a generic visual-decay pool (ground puffs ride it). Blood is a **fixed pool of 72 sprites** (`effects.js`, each with a velocity so it SPRAYS + falls — `spawnBloodBurst(x,y,z,dirx,dirz,n,power,spread)`; `spread` = cone width, `6.283` = 360° for explosions), explosion flashes a **fixed pool of 3 lights** — keep point-light counts constant (no mid-game shader recompiles). New entity type = mesh + array + update block + `resetGame` cleanup.
- **GORE (`entities/gore.js`, 2026-07-11):** robots don't vanish on death — `killRobot(i, {cause,dirx,dirz})` **splices the robot from the live `robots` array FIRST** (so the corpse is inert: no AI/claw/body-block/bullet-hit and it doesn't hold up the survival wave-clear check), awards score, then **hands the robot's own mesh to `spawnCorpse`** (a topple animation about the feet → brief lie → fade [materials flipped `transparent`, no recompile] + slight sink → dispose ~1.7 s) and flings **gibs** (`gibRobot` hides limb pivots on the corpse and throws matching limb-coloured chunks). A `spawnBloodBurst` + a ground `spawnBloodDecal` fire too. **`cause: 'explosion'` = extra-gory dismemberment** (2× omnidirectional [360°] blood bursts ~52 sprites, the 5 limb gibs PLUS ~10 extra flesh chunks, and ~9 ground blood pools SCATTERED around the blast + one per gib landing); bullet/melee = a spurt + occasional single limb. **Gibs and ground blood decals are FIXED POOLS** (`GIB_COUNT` 64 / `DECAL_COUNT` 44) created in `initGore(scene)` and added to the scene hidden — so preload's `renderer.compile` (which traverses hidden objects) compiles their shaders → no mid-game recompile/hitch. `updateGore(dt)` runs in `updateGame` after `updateRobots`; `resetGore()` in `resetGame` disposes corpses + hides the pools. Gib/decal geometries + the decal texture are SHARED (never disposed); corpse materials are the robot's per-instance ones (disposed with the corpse).
- **Robots:** procedural **stylized low-poly** humans (`buildHumanRobot`, 6 profession variants; **overhauled 2026-07-11 from cubic Minecraft/Roblox blocks to rounded forms — tapered `CylinderGeometry` limbs + `ellipGeo` ellipsoids for head/hands/feet, a flattened tapered torso, a shoulder-yoke ellipsoid, a short neck, and sunken dark eye sockets**; shared `ZG` geometries [built once via `geo.scale()` baking] + per-instance materials; never dispose `ZG`). The head is now a single skin ellipsoid + 2 eye meshes (the old box-face-texture `robotFaceTex`/`drawRobotFace` were removed); `disposeRobot` still handles array materials defensively. **The animation rig contract is UNCHANGED** — same pivot groups (`inner`, hip/knee/shoulder, `headG`) at the same positions, so `animateRobotRig` is byte-identical. **Behavior variants** (2026-07-07, `CFG.robot.variants`): runner (fast, low hp, 25% smaller [`scale` 0.75], pale-yellow skin), brute (tanky, hard-hitting, ×1.25, charcoal/blackish skin), exploder (reddish skin + subtle red glow; queued blast on death via `pendingBooms` — NEVER call `explodeAt` inside the robot loop, it splices the same array; the queue runs after the loop and hurts the player in `boomRadius`), plus the stage-2 **boss** (`CFG.campaign.boss`: grenades deal the reduced `boss.grenadeDamage` instead of the full `grenade.damage`, tested by `z.kind === 'boss'` in effects.js). Scaled robots MUST get `reachMul` from `reachForScale(scl)` — their scaled body-block radius otherwise pushes the player out of their own claw reach. Rig animated by direct `rotation.x` on pivots (`animateRobotRig`). **No damage feedback by design** (player must not see how hurt a robot is). **Damage is a numeric model (rebalanced 2026-07-07):** each bullet carries its weapon's `damage` (rifle 30 / pistol 20 / shotgun 5-per-pellet), robots have real HP (base 60, variant `hpMul` unchanged), grenades deal `grenade.damage` 100, player HP is 100. **No headshots (removed with the top-down pivot 2026-07-11):** every bullet deals its flat `damage` wherever it lands. **The bullet-vs-robot hit test is HORIZONTAL (2D, xz-plane) — the y dimension is intentionally dropped** because aiming is horizontal-only and all entities stand on the ground; a 3D sphere test let short hitboxes (esp. the scale-0.75 runner, center y≈4.5 / top≈8.25) slip UNDER the barrel-height (~7.65) bullet path and be unhittable point-blank. The hitbox is effectively a vertical cylinder of radius `bodyHitRadius` 5 × `z.scl` (runner 0.75 / brute 1.25 / boss 2.4); the swept segment (prev→current, projected to xz, spawn frame from the player center) still prevents point-blank/low-fps tunneling. Survival robots jump the fence (two-phase `jumping`→`chasing`, arc reused for the fountain vault); campaign robots start `'idle'` (breathing, shootable, solid) and wake by distance (`activateMeters`×7) + stage-1 grid LOS, or when shot. **Chasing uses the pathfinder** (`utils/pathfind.js`, added 2026-07-06): each scene bakes a nav-grid once (survival: Monas+trees blocked, fountain deliberately walkable so the vault still triggers; stage 1: half-cell 7-unit grid — walls from the floor-plan grid PLUS furniture/stair blockers baked via resolve, so robots route around desks instead of shoving them; stage 2: road corridor with medians/wrecks baked via resolve — paths go through median gaps). `navAim` checks grid-LOS **every frame** (cheap) and returns a shared `{x, z, direct}` — direct chase the instant the player is visible, waypoint steering the instant the line is blocked (no timer lag; only `findPath` is rate-limited by `CFG.robot.repathSec`, forced when stuck > `stuckSec`); waypoints also advance early when the next one is visible (runtime string-pull). Movement runs along a turn-rate-limited heading (`turnToward`, `CFG.robot.turnRadPerSec`) so robots curve smoothly instead of snapping between waypoints. Path failure/off-grid falls back to straight chase. Robots no longer wedge against walls/trees/Monas. Claw attack: `CFG.robot.clawDamage` 5 per swipe (base/walker), `clawCooldownSec` 1 per robot (variants/boss/Monas carry their OWN absolute claw, NOT a multiple of base — runner 20, brute 40, exploder 20, boss 60; against the player's 100 HP); attacking robots stand still (walk cycle damps out).
- **Weapons:** FOUR weapons via the `WEAPON_DEF` table in weapons.js (all per-weapon visual constants live there): `'rifle'` (displayed **"Assault Rifle"**, modeled after the Pindad SS2-V2 — all-black, carry handle with ADS sight channel, ribbed handguard, skeleton folding stock), `'pistol'`, `'shotgun'` (pump-action, box mag), and **`'launcher'`** (displayed **"Grenade Launcher"**, added 2026-07-11 — a fat 40mm break-barrel; Survival-shop-exclusive, 2000 pts). **The Grenade Launcher fires ONE explosive round per trigger pull (shotgun-like `fireDelayMs` 900, `maxAmmo` 50) that EXPLODES ON IMPACT** — the round is a bullet with `b.explosive`/`b.explodeR` (the Mk2 mesh via `buildGrenadeMesh`, slower `roundSpeed`, no tracer stretch); it detonates on hitting a robot (`robots.js` hit loop) or a wall/Monas (`bullets.js` `hitWall`) — **never on pure life-expiry** — by QUEUEING an `explodeAt` via `queueBoom(x,y,z,r,hurtPlayer,playerDmg)` (generalized `pendingBooms` in robots.js, processed after the robot loop = no reentrant splice; launcher booms are `hurtPlayer:false` = friendly). Same AoE as the old grenade (`CFG.grenade.killRadius`+3.5, `grenade.damage` 100). The exploder self-boom also routes through `queueBoom` (`hurtPlayer:true`). **Model overhaul 2026-07-10 (shapes only — every muzzle point, weapon position, and ADS sight height is at its old calibrated coordinate; do NOT move them):** rifle = split upper/lower receiver + curved banana mag + ejection-port/selector/trigger/buttpad details; pistol = composite slide with serrations and the SIGHTS RIDING THE SLIDE (same absolute offsets — they move with reload pulls now), full trigger-guard loop, hammer, grip panels; shotgun = sighting rib, barrel clamp, ribbed pump, comb+recoil-pad stock. **`pistolSlide`, `pistolMagMesh`, `sgPump`, `sgMagMesh` are now THREE.Groups** (were Meshes) — the reload rigs only write `.position`/`.rotation` so the animation contract is unchanged; keep it that way. **Hands:** `wrapFinger` helper builds fingers as tangent arc segments that genuinely wrap the handguard/pump/grips. **Rifle/shotgun left hand = OVERHAND grip matching the user's CS-style reference (2026-07-10): back of the hand faces the camera on the near side, four fingers drape OVER the top of the handguard/pump from near to far, thumb tucks along the near-bottom toward the muzzle, bare-skin wrist below — don't flip it back to an under-grip.** Fingerless-glove look: `wrapFinger`'s optional `tipMat` renders the last 2 finger segments in a shared `skin` Phong material (same shader program as the glove — warm-up unaffected); pistol support hand is thumbs-forward along the frame. The left forearm placeLimb endpoints in `updateWeaponVisuals` target the overhand wrist offset (−0.45, −0.75/−0.65, +0.8 from the hand group). (Shotgun firing: `pellets: 10` bullets per trigger pull with extra `pelletSpread`; the launcher fires 1 explosive round; rifle/pistol have no `pellets` key = single bullet). **Each weapon has a `damage` in `CFG.weapons.<w>` (rifle 30 / pistol 20 / shotgun 5-per-pellet); the value is stamped onto every bullet at spawn (`b.damage`) so the hit-test in robots.js applies the right amount even after a weapon swap (fallback `CFG.weapons.bulletDamage`).** **MAGAZINE SYSTEM REMOVED (2026-07-11):** each weapon is ONE ammo pool — `player[w].ammo`, capped by `CFG.weapons.<w>.maxAmmo` (**rifle 500 / pistol 150 / shotgun 300**), starts FULL, refilled by ammo drops (`+CFG.weapons.<w>.ammoPickup` per owned weapon, the old 'mag' drop type/mesh reused) and the shop's Replenish All Ammo (sets to max).

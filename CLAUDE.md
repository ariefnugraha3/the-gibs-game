This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Overview

Browser FPS: "Gibran vs Zombie 3D (DOOM FPS)", Three.js r128. **All user-facing UI text MUST be in English — permanent user rule (2026-07-05), no exceptions, applies to every future change.** Code comments are in Indonesian. **Two game modes** picked on the `#modeSelect` screen at load (plus an EASY/NORMAL/HARD difficulty row — `applyDifficulty` recomputes CFG from the pristine `CFG_BASE`, per-difficulty high scores): **Survival** (round-based waves — you start with **only a pistol + 3 mags**, wave 1 spawns 15 zombies [`zombiesPerWaveBase` + `Step`·(n−1)], clearing a wave shows "WAVE CLEARED" + a `shopCountdownSec` countdown, then the between-wave **Field Shop** auto-opens [a clickable mouse-driven menu — buy ammo/grenade/health refills, a **Medkit** (held, press 4 to use for +70% HP), Heal/Strengthen Monas, the **Radar** (Survival starts without it), and the Shotgun & Assault Rifle (max 2 weapons — buying a 3rd prompts which to replace); click an item then its **Buy** button, then **Start Next Wave** → "Are you ready?" confirm]; **zombies prioritize the Monument** — every zombie attacks Monas by default and only aggros onto the player when the player is within `playerAggroMeters` (20 m) of that zombie (returning to Monas past it), `monasHp` 0 = "THE MONUMENT HAS FALLEN"; score = shop currency; every 5th wave triggers a fog/blackout event) and **Campaign**, a linear 3-stage mode — Stage 1 "Gedung Terbengkalai (Lantai 3)" (indoor 60×60 m grid building), Stage 2 (ruined-Jakarta highway, ends with a boss), Stage 3 (night-time Monas park, reusing the survival world). Campaign zombies lurk motionless until seen/approached.

> **Desktop/Steam port:** the plan to wrap this into a Windows `.exe` (Electron) and ship on Steam/itch lives in [STEAM-DESKTOP-PLAN.md](STEAM-DESKTOP-PLAN.md).
>
> **Gameplay roadmap:** the agreed feature backlog (zombie behavior variants, stage-2 boss, survival shop, music, shotgun, etc.) with per-item implementation notes, pitfalls, and status lives in [IMPROVEMENT-PLAN.md](IMPROVEMENT-PLAN.md) — read it before starting any new gameplay feature, and update its status table when you finish one.

## Architecture (refactored 2026-07-05 into ES modules)

The old single-file `index.html` was split into **ES modules under `src/`** with a **scene system**. There is still **no build step, no npm dependencies, no framework** — plain static files; Three.js r128 + post-processing examples come from CDN `<script>` tags as the global `THREE` (modules do NOT import it). `package.json` is metadata-only (`"type": "module"` for Node tooling).

**→ The complete module catalog, every export, the scene interface contract, the config key table, and the how-to-add-a-stage recipe live in [MODULES.md](MODULES.md). Read it before hunting through files.**

Load-bearing rules (details in MODULES.md):

- **Scenes:** `menu` (DOM-only) → `survivalScene` or `campaign stage1Scene → stage2Scene`, managed by `core/sceneManager.js` (`setScene`/`activeScene`). All mode-specific behavior (collision walls, ground height, bullet blockers, grenade bounds, zombie movement AI, radar landmarks, HUD status, drop clamping, win check, restart policy) is behind `activeScene.*` hooks — **shared systems contain zero mode if-else**. Adding a campaign stage = one new file in `src/scenes/campaign/` + wiring per the MODULES.md recipe.
- **Both campaign stage worlds coexist in one THREE.Scene** ~26 km apart (building at x≈30000); `camera.far` 4000 + zombie distance culling hide the far one. Campaign restart (death) always re-enters stage 1, which re-places both stages' zombies + supplies (arrays are cleared by `resetGame` first). Every campaign zombie carries `z.stage`; stage transition disposes the previous stage's zombies silently.
- **All gameplay-tunable numbers live in `config/gameplay.json`**, loaded into `CFG` (`core/config.js`) before the game starts. Read `CFG.x.y` inside functions only (never at module top level). New tuning values go in the JSON, not as code constants. Visual-only values (FOV, animation amplitudes, colors) intentionally stay in code. Reload hand-animation keyframes auto-scale to `reloadMs`.
- **Cross-module state = live ESM bindings**: the owning module exports `let` + setters (`isPaused`/`score` in `core/state.js`, `stamina`/`sprintingNow`/`crouchedNow` in `entities/player.js`, `currentWeapon`/`isAiming`/`switchAnim` in `entities/weapons.js`, `scene`/`camera`/`renderer` in `core/renderer.js`). Circular imports are fine and used (weapons↔player, zombies↔effects, stage1↔stage2, game↔input) **as long as bindings are only used inside functions**.
- **`updateGame()` block order in `core/game.js` is a contract** (matches the old `update()`): scene updateMode → weapon timers → player movement → weapon state → shooting → grenades → explosions → blood → drops → bullets → zombies → win check. Bullets must move before the zombie sweep hit-test.
- The **camera is the player** (no player mesh; guns are camera children). Decor animation (`world/decor.js` registry + weapon visuals) runs every frame **even while paused**.

## Running

**A static HTTP server is now mandatory** (ES modules + `fetch` of the JSON config are blocked on `file://`; the page shows an English error explaining this if opened directly):

```
python -m http.server 8000      # then open http://localhost:8000
```

Internet required for the Three.js CDN scripts. No build, no install. For syntax checking: `node --check src/<file>.js` (works thanks to `"type": "module"`). A headless test harness pattern (Node + stubbed THREE/DOM driving the real modules) is described at the end of MODULES.md.

## Key mechanics and conventions

- **Collisions are distance checks, not physics.** Zombie-hit, pickup, and blast use `distanceTo` radii. **Bullet-vs-zombie is a swept-segment check** (`segPointDist2`, prev→current position; spawn-frame segment starts at the player's EYE so point-blank never tunnels). Survival: Monas is a solid AABB resolved PER-AXIS (`resolveMonas` in world.js — `MONAS_HALF` 22, base 44×44; hug-and-slide, NOT a full revert, so the player never sticks — mirrors the campaign wall fix), solid trees/fountain cylinders (`resolveObstacles`), fence clamp. **`bulletBlocked` follows the real Monas silhouette height-by-height** (base 44 low → thin obelisk ~16 at eye height, nothing above y 64) so bullets pass through the empty air beside the obelisk and hit zombies behind — do NOT revert it to a flat 44-wide column. Campaign: walkable-union tests (`stage1Walk` grid / `highwayWalk`) with per-axis `slideWalk` (hug walls, don't stick) + rotated-AABB `resolveBlockers` (medians/wrecks/furniture; standable tops are walkable). Zombie bodies are solid to the player only (push-out at `CFG.zombie.bodyBlockRadius` 7.5, deliberately < claw-stop 8.0 so an attacking zombie doesn't shove the player every frame).
- **Frame-rate independence:** `dt` (clamped 0.05) and `step = dt*60`. Multiply motion by `step`; decrement timers by `dt`. Fire-rate/reload use real `Date.now()` time.
- **Entity model:** five arrays in `core/state.js` — `bullets, zombies, grenades, explosions, drops` — iterated backwards for splicing. `explosions` doubles as a generic visual-decay pool (ground puffs ride it). Blood is a **fixed pool of 14 sprites** (`effects.js`), explosion flashes a **fixed pool of 3 lights** — keep point-light counts constant (no mid-game shader recompiles). New entity type = mesh + array + update block + `resetGame` cleanup.
- **Zombies:** procedural blocky humans (`buildHumanZombie`, 6 profession variants, shared `ZG` geometries + per-instance materials — `disposeZombie` handles the head's material array; never dispose `ZG`/face texture). **Behavior variants** (2026-07-07, `CFG.zombie.variants`): runner (fast, low hp, smaller), brute (tanky, hard-hitting, ×1.25), exploder (green glow; queued blast on death via `pendingBooms` — NEVER call `explodeAt` inside the zombie loop, it splices the same array; the queue runs after the loop and hurts the player in `boomRadius`), plus the stage-2 **boss** (`CFG.campaign.boss`, `z.noInstakill`: headshots use the boss's own `headshotDamageMul` 3× instead of the regular 2×; grenades deal the reduced `boss.grenadeDamage` instead of the full `grenade.damage`). Scaled zombies MUST get `reachMul` from `reachForScale(scl)` — their scaled body-block radius otherwise pushes the player out of their own claw reach. Rig animated by direct `rotation.x` on pivots (`animateZombieRig`). **No damage feedback by design** (player must not see how hurt a zombie is). **Damage is a numeric model (rebalanced 2026-07-07):** each bullet carries its weapon's `damage` (rifle 30 / pistol 20 / shotgun 5-per-pellet), zombies have real HP (base 60, variant `hpMul` unchanged), grenades deal `grenade.damage` 100, player HP is 100. A **headshot** (`CFG.zombie.headHeight` 11.5, radius `headshotRadius` 1.6) deals `CFG.zombie.headshotDamageMul` (2×) the bullet's damage — **no longer an instant kill** (boss uses its own 3×). A weapon may carry its own headshot multiplier (`CFG.weapons.<w>.headshotMul` → `b.headshotMul`, wins over the zombie/boss default): **shotgun pellets headshot at 4×**. Hitboxes are deliberately slim (`bodyHitRadius` 5, `headshotRadius` 1.6, narrowed head/torso geometry — height preserved) so precise aim matters. Survival zombies jump the fence (two-phase `jumping`→`chasing`, arc reused for the fountain vault); campaign zombies start `'idle'` (breathing, shootable, solid) and wake by distance (`activateMeters`×7) + stage-1 grid LOS, or when shot. **Chasing uses the pathfinder** (`utils/pathfind.js`, added 2026-07-06): each scene bakes a nav-grid once (survival: Monas+trees blocked, fountain deliberately walkable so the vault still triggers; stage 1: half-cell 7-unit grid — walls from the floor-plan grid PLUS furniture/stair blockers baked via resolve, so zombies route around desks instead of shoving them; stage 2: road corridor with medians/wrecks baked via resolve — paths go through median gaps). `navAim` checks grid-LOS **every frame** (cheap) and returns a shared `{x, z, direct}` — direct chase the instant the player is visible, waypoint steering the instant the line is blocked (no timer lag; only `findPath` is rate-limited by `CFG.zombie.repathSec`, forced when stuck > `stuckSec`); waypoints also advance early when the next one is visible (runtime string-pull). Movement runs along a turn-rate-limited heading (`turnToward`, `CFG.zombie.turnRadPerSec`) so zombies curve smoothly instead of snapping between waypoints. Path failure/off-grid falls back to straight chase. Zombies no longer wedge against walls/trees/Monas. Claw attack: `CFG.zombie.clawDamage` 5 per swipe (base/walker), `clawCooldownSec` 1 per zombie (variants/boss/Monas carry their OWN absolute claw, NOT a multiple of base — runner 20, brute 40, exploder 20, boss 60; against the player's 100 HP); attacking zombies stand still (walk cycle damps out).
- **Weapons:** THREE weapons via the `WEAPON_DEF` table in weapons.js (all per-weapon visual constants live there): `'rifle'` (displayed **"Assault Rifle"**, modeled after the Pindad SS2-V2 — all-black, carry handle with ADS sight channel, ribbed handguard, skeleton folding stock), `'pistol'`, and `'shotgun'` (pump-action, box mag; `pellets: 7` bullets per trigger pull with extra `pelletSpread` — regular weapons have no `pellets` key = single bullet, byte-identical old path). **Each weapon has a `damage` in `CFG.weapons.<w>` (rifle 30 / pistol 20 / shotgun 5-per-pellet); the value is stamped onto every bullet at spawn (`b.damage`) so the hit-test in zombies.js applies the right amount even after a weapon swap (fallback `CFG.weapons.bulletDamage`).** Different fire delay/reload/SFX/rig. **Inventory is 2 ORDERED weapon slots (`player.weapons`, max `CFG.weapons.maxWeapons` 2, distinct types) + grenade + medkit (2026-07-08 overhaul):** key **1** = slot 0's weapon, **2** = slot 1's weapon, **3** = equip GRENADE (`grenadeMode`, see Grenades), **4** = use Medkit, **Q** = swap between the two weapon slots; 0.5 s switch (model swap at midpoint, cancels reload CS-style). `player.owned` is DERIVED from `player.weapons` via `syncOwnedFromWeapons()` (state.js) — the ordered array is the source of truth for slot→key mapping, `owned` for "do I have this". **Loadout per mode (`configurePlayer`): Survival starts `['pistol']` only (buy the rest in the shop); Campaign starts `['rifle','pistol']` (shotgun is Survival-shop-exclusive).** Starting weapon (`applyStartLoadout`/`pickStartWeapon`) = `player.weapons[0]`; `lastWeapon` (Q target) = the other slot. `trySwitchKey` ignores empty slots. After a shop buy/replace, `refreshOwnedWeapon()` re-points `currentWeapon`/`lastWeapon` if the active weapon was swapped out. Reload duration = `reloadMs × player.reloadMul` (shop upgrade), stored in `player.reloadDurMs` which the reload keyframe rigs read — timer and animation stay in sync. Accuracy: camera-space spread cone `(spreadBase + gunHeat·spreadBloom) × acc × movePen`; `acc` = ADS 0.75 × crouch 0.75; movePen walk 1.25 / sprint 1.5; camera kick via the same YXZ euler as mouse-look. `gunRecoil` is visual-only. Mags cap at `maxMags` 5 per weapon; mag pickups refill all **owned** weapons.
- **Grenades:** ballistic + 3 s fuse (never on-impact), real-seconds gravity, bounce/roll with friction, solid-when-landed (`nudgeGrenade` position-ejects and caps added speed at `pushSpeed` 22 — natural shove, not a kicked ball). Shared Mk2 mesh resources — never dispose. Rolling SFX once per grenade, ≤90 units, distance-scaled. **Throw mechanic (reworked 2026-07-08 — G key removed):** press **3** to EQUIP a grenade (`equipGrenade` in weapons.js → `grenadeMode`, gun holstered, a gloved-hand-holding-Mk2 rig `grenadeHandMesh` raised; blocks shooting/reload/melee/ADS). Then **left-click = throw FAR** (`throwSpeed`/`throwUpward`), **right-click = throw NEAR** (`throwNearSpeed`/`throwNearUpward`) — input.js routes clicks to `throwEquippedGrenade(range)`. The throw is a timed animation (windup → release at ~0.30 s → follow-through); `spawnGrenade(range)` in grenades.js does the physics at the release point. After the last grenade is thrown, auto-holster to `currentWeapon`; pressing 1/2/Q also holsters. `player.grenades` cap `grenade.max` 5.
- **Stamina** (`CFG.stamina`): sprint 10/s (only while actually moving — `sprintingNow` also drives footstep cadence, bob, spread penalty), ADS 3/s, melee 15/swing, regen 12/s; empty → `staExhausted` locks sprint+ADS until ≥ 20 (hysteresis).
- **Movement:** WASD combined into one vector, normalized against raw input length (diagonals not faster); backpedal ×0.7, strafe ×0.8; crouch ×0.5 **replaces** ADS ×0.7 (never stacked); sprint ×1.7. Jump `jumpVelocity` 33 / `gravity` 70 (real seconds); `eyeHCur` = eye height lowered by crouch — every feet↔camera conversion uses it, and the crouch block computes feet with the previous frame's value (no phantom-fall SFX).
- **Coordinates are large-scale**: 1 m ≈ 7 units (`CAMP_M`); eye height 11.4; park half-extents 620×340; keep new geometry proportional.
- **HUD:** bottom-left `#barsHud` (HEALTH bar — always red, CSS-only color, JS writes width only; STAMINA bar — green→orange, red = exhausted). Bottom-right `#statsHud` (Score/Ammo — Ammo shows the active weapon, or a "Grenade — LMB Far / RMB Near" hint in `grenadeMode`). **Right-side `#inventoryHud`** (2026-07-08): a vertical 4-slot panel built by `updateInventory()` in hud.js — `1`/`2` = weapon slots (`player.weapons`, name from `WEAPON_DEF`, `—` when empty), `3` = `Grenade ×N`, `4` = `Medkit ×N`; the active slot (held weapon or `grenadeMode`) gets `.active` (gold), empty/0 slots get `.dim`. Slot refs via `invSlots` in dom.js. **Radar is top-left** (moved 2026-07-05): canvas-drawn flat dark bg + range rings + FOV cone + world-N marker + glowing blips (gradient bg & rotating sweep removed by user request 2026-07-05 — don't re-add); projection via `radarProject` in `core/hud.js` (heading-up: px = world-right, py = −world-forward — the old formula was 180°-flipped, don't reintroduce it); landmarks passed `clampEdge=true` pin to the rim as direction pointers when out of range. **Gated by `player.hasRadar`** (2026-07-08): Survival starts WITHOUT the radar and buys it in the shop (`Radar` item); `drawRadar` skips and `updateUI` hides the `#radar` canvas until owned. Campaign/other modes have it from the start. `#waveText` top-center via `activeScene.hudStatus()`. `#pickupFeed` mid-left (`showPickup`), full-item pickups are skipped with a grey "... already full" line (1.2 s rate limit). **`#crosshair` stays permanently hidden via CSS** (user request) — the element and all JS writes to it are intact so shooting/recoil stay byte-identical; don't "clean up" and don't re-show.
- **Graphics quality**: 5 tiers on the start screen only (`initQualityUI`, localStorage `gibsQuality`; hidden permanently after first pointer lock). Runtime-safe knobs (pixel ratio, shadow size, bloom, composer on/off); tier 4 disables shadows (one recompile, safe because start-screen-only). Don't call `applyQuality` mid-game.
- **Rendering:** sRGB + ACES (exposure 1.1 = brightness knob) + PCFSoft shadows following the player (`updateShadowFollow`); composer RenderPass → UnrealBloom → GammaCorrection → FXAA with null-fallback if CDN fails; hard-bloom materials set `toneMapped: false`; instanced meshes set `frustumCulled = false` (r128 doesn't merge instance bounds); city/facade recipes shared in `world/facades.js`; all textures procedural canvas (`makeTexture`/`makeNormalMap` — normal maps stay linear).
- **Input safety:** `requestLock()` swallows Chrome's cooldown rejection and calls `enterImmersive()` — fullscreen + **Keyboard Lock** (`LOCK_KEYS`, deliberately excluding Escape) so Ctrl+W/Ctrl+R can't kill the tab mid-game (Chromium; `beforeunload` guard as Firefox/Safari fallback). `releaseInputs()` clears keys/aim/crouch on unlock/blur/reset. Crouch: C toggle + Left-Ctrl hold (keyup clear ungated).
- **SFX:** always via `playSFX` (per-clip pool of 8). Zombie footsteps: ONE global timer plays the nearest moving chaser's step (~0.55 s, ≤120 units) — never per-zombie. Weapon audio in `weapons.js`: firing picks `sfxShoot` (rifle) / `sfxPistol` / `sfxShotgun` per `currentWeapon`; `startSwitch` plays `sfxSwitch`; a dry-fire `sfxEmpty` "click" plays once per trigger-pull (gated by module `emptyReady`, re-armed on mouse-up) when the trigger is pulled with ammo AND mags both at 0. `zombie-spawn.mp3`, `shotgung-cocking.mp3`, and `assets/visuals/zombie.glb` exist on disk but are unused.
- **`resetGame()`** (core/game.js) clears every entity array + pools (+ `resetZombiesFx()` exploder-boom queue + `resetStats()`), resets player/weapons/stamina/shop-upgrades via module reset fns, then re-enters the restart scene (`activeScene.restartScene()` — campaign always stage 1). Keep it in sync when adding entity types.
- **Survival is round-based (overhauled 2026-07-07):** `wave.phase` state machine in `survival/index.js` — `'fighting'` (spawn a FIXED per-wave budget `wave.toSpawn`, capped on-field by `maxConcurrent`) → wave cleared when `toSpawn===0 && zombies.length===0` → `'cleared'` (show "WAVE CLEARED" + `shopCountdownSec` countdown in `hudStatus`) → `'shopping'` (auto-`openShop()`; SPACE = `startNextWave()` → `startWave(num+1)`). Weapon inventory: `player.weapons` ordered slots (survival starts `['pistol']`, max 2; `owned` derived via `syncOwnedFromWeapons`); the Shotgun & Assault Rifle are bought in the shop, which appends to a free slot (full loadout) or, when both slots are full, prompts a REPLACE chooser. Field shop (`survival/shop.js`, **clickable mouse-driven MODAL** — game is PAUSED and the pointer released while it's open, the world still renders behind it; auto-opens between waves): a two-column menu — left = item list (hover → detail), right = name/description/price + **Buy** button, bottom-left = Score, bottom-right = **Start Next Wave** button. Items (data-driven `catalog()`; **buying is via the Buy button ONLY** — clicking a list row just selects/previews it, never buys — `shopPurchase(id)` does the transaction; currency = score): Replenish All Ammo (owned weapons only), Replenish Grenades, **Replenish Health** (`healthCost`; instant full heal to 100%), **Medkit** (`medkitCost`; a HELD medkit — max `CFG.player.maxMedkits` 1, `player.medkits`; USE with key 4 in the field for +70% HP), **Heal Monas** (+25% of Monas max HP), **Strengthen Monas** (raises Monas max HP to the next tier in `CFG.survival.strengthenMonasStages` [3000→4500→6000, last = ceiling] + heals by the increase; scene-local `monasStage`; shows "Maxed" at the ceiling; **the item description deliberately hides the HP numbers** per user request — it only says it reinforces the Monument), **Radar** (`radarCost`; sets `player.hasRadar` → the top-left minimap; Survival starts WITHOUT it, shows "Owned" once bought), Buy Shotgun, Buy Assault Rifle (append to a free weapon slot + full loadout; **buying a weapon while both slots are full opens a REPLACE chooser** — `shopPurchase` returns `'choose-replace'`, then a click on `shopReplaceWeapon(oldW)` swaps the new weapon into that slot and deducts the score, `shopCancelReplace` aborts). Buys are rejected (score kept) when full/owned/insufficient. Clicking **Start Next Wave** (or SPACE/Enter via scene `shopKey`) first shows an **"Are you ready? Yes / No"** confirm (`requestNextWave`/`confirmNext` in shop.js); confirming (click Yes, or a second SPACE) actually starts the wave. Buying is Buy-button-only. `input.js` keys off `activeScene.shopActive()`: while the shop is open it swallows all gameplay keys and, on the deliberate pointer-unlock, pauses WITHOUT showing the pause blocker (cursor freed for the menu); `startNextWave()` closes the shop then calls `requestLock()` to re-lock + resume. `monasHp`/`monasMaxHp` are scene-local (max re-seeded from `CFG.survival.monasMaxHp` on `enter()`, so Strengthen resets each run). Mag drops refill only `player.owned` weapons. **Monas HP objective (priority target, reworked 2026-07-07):** every survival zombie attacks Monas by default — per-frame `atkMonas = monasHp>0 && distToEye > CFG.survival.playerAggroMeters × CAMP_M` (20 m → 140 units); it only chases the player when the player is within that aggro radius, and returns to Monas once the player leaves it (no static `z.target` / `monasAttackerChance` anymore). Monas attackers gnaw the AABB edge and return `{}` from `zombieAI` so they never claw the player. **Scoring** (`CFG.zombie.score`, applied in `killZombie(i, headshot)`): normal kill 100 / headshot 150, special-variant kill 150 / headshot 200 (boss = `boss.score`); melee & grenade kills pass headshot=false. Wave events (fog animates the ALWAYS-EXISTING `scene.fog` near/far, blackout scales light intensities — both computed from `LIGHT_PRESETS.outdoor`, restored exactly on end/enter; never create/remove fog at runtime = shader recompile). Damage direction indicator: one reused `#hitDir` wedge rotated by `attackerAngle()`; end-of-run stats (`stats` in state.js, per-bullet accuracy) fill `#goStats` on the game-over screen.

## Editing guidance

- Keep the project a **static, buildless site**: no bundlers, frameworks, or npm dependencies (package.json stays metadata-only). It must run from a plain HTTP server.
- Follow the architecture rules at the top of [MODULES.md](MODULES.md) (scene hooks instead of mode if-else; CFG for tunables; live-binding state ownership; updateGame block order) and update MODULES.md whenever modules/exports/config keys change.
- Preserve existing controls (WASD move, Shift sprint, mouse look, left-click fire, right-click toggle ADS, **1/2 = weapon slots, Q = swap weapon, 3 = equip grenade [then left-click = throw far / right-click = throw near], 4 = use medkit**, F melee, R reload, C toggle crouch / Left-Ctrl hold crouch, Space jump / restart / Start-Next-Wave in the survival shop) and core mechanics unless the user asks for gameplay changes.
- If you edit the stage-1 room layout, re-verify grid connectivity (BFS from `S1_START` over floor cells) and that no furniture blocker parks in front of a doorway.

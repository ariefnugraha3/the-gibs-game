# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Overview

Single-page browser game: "Gibran vs Zombie 3D (DOOM FPS)". A first-person Three.js shooter where the player defends Monas (Jakarta's national monument) against waves of zombies. UI text is in Indonesian; code comments are in Indonesian.

## Running

No build system, package manager, or tests. Open `index.html` directly in a browser, or serve the folder with a static HTTP server (preferred — required for the `.mp3` SFX to load reliably and for PointerLock):

```
python -m http.server 8000      # then open http://localhost:8000
```

Three.js r128 and the post-processing example scripts are loaded from CDNs (`<head>` scripts), so an internet connection is required.

## Architecture

Everything lives in [index.html](index.html) — inline CSS, DOM overlay, and the full game in one `<script>`. There are no modules; all functions and state are top-level globals. The only assets are SFX in `assets/sounds/` (`*.mp3`); all visuals (textures, zombies, weapons) are procedural. `assets/visuals/zombie.glb` still exists on disk but is **no longer used**.

- **Entry point:** `init()` builds the scene then `animate()` starts the render loop. Both are called immediately after global state declarations (~line 523).
- **Opening cutscene:** a self-invoking `initCutscene()` IIFE (bottom of the script) runs a pure-DOM/CSS 4-slide slideshow (`#cutscene` overlay) before play. It's independent of the Three.js loop — `finish()` just hides the overlay to reveal `#blocker`, whose click requests PointerLock. No game state is touched here.
- **Environment is decorative:** `createSky` (canvas-gradient sky dome with stars/smoke + moon and additive halo), `createCity` (two building rings at radius ~820–1380, outside the park + ring road, with rubble piles at building feet), and `createEmbers` (a `THREE.Points` ash field) build the apocalyptic backdrop. None of it has collision. Ambient animation lives in `updateEmbers` (upward drift + wind sway, recentered on the player) and `updateDecor` (runs every frame even while paused: shadow camera follows the player, Monas flame + burning-building flicker, fountain jets pulse, muzzle-flash sprite fade).
- **Rendering pipeline:** sRGB output + ACES filmic tone mapping (`toneMappingExposure` is the single brightness knob) + PCFSoft shadow maps (2048px, ortho ±320 around the player; `dirLight.target` is in the scene so `updateDecor` can move both). Casters: zombies, trees, fence, Monas, fence rails; the distant city does not cast. **Keep the point-light count constant** — explosion flashes reuse the 3-light `explosionLights` pool (intensity 0 when idle) so Three.js never recompiles shaders mid-game; `clearArray`/the explosion loop set `o.light.intensity = 0` instead of removing lights.
- **Post-processing:** `composer` chain RenderPass → UnrealBloomPass (0.45/0.4/0.72) → GammaCorrectionShader (the RTs are linear, so this pass does the sRGB conversion) → FXAAShader (`setFxaaRes()` must be called on resize). Eight extra CDN example scripts feed it; if any fail to load, `composer` stays `null` and `animate()` falls back to plain `renderer.render`. Materials that must bloom hard set `toneMapped: false` (tracers, explosion core/shock, flame glow, muzzle sprite, fire sprites).
- **Instancing:** city buildings (normal / burning / rubble), park trees (trunk + 2 canopy layers), and fence posts are `THREE.InstancedMesh` with per-instance colors — a handful of draw calls instead of ~500. All instanced meshes set `frustumCulled = false` because r128 doesn't compute merged instance bounds (otherwise they vanish when the unit geometry leaves the frustum).
- **Textures are procedural:** `makeTexture(w, h, drawFn, repX, repY)` + `speckle()` generate every color map (grass blades, asphalt + cracks + lane dashes, radial plaza paving, marble, wood grain, building facades/lit windows, water ripples, muzzle star-flash, moon craters, fire/halo/ember dots) on canvas — no image files. `makeNormalMap(w, h, heightDrawFn, strength)` (+ `noiseHeight`) bakes normal maps from canvas heightmaps; normal maps stay linear (no sRGB), and in r128 the UV transform comes from `material.map`, so a shared normal map follows each material's diffuse repeat. Canvas color textures are flagged `sRGBEncoding`.
- **Game loop:** `animate()` → `requestAnimationFrame` → `update(dt, step)` → `renderer.render`. `update()` early-returns when `isPaused` or `isGameOver`, so all gameplay (movement, shooting, AI, collisions) only advances while pointer-locked and alive.
- **Frame-rate independence:** `animate()` derives `dt` from a `THREE.Clock` (clamped to 0.05 to absorb tab-switch spikes) and `step = dt * 60`. Speed constants are calibrated at 60 fps and multiplied by `step`; per-frame timers (drops, explosions, waves) decrement by `dt` in seconds. When adding motion, multiply by `step`; when adding a countdown, subtract `dt`.
- **Entity model:** five arrays — `bullets`, `zombies`, `grenades`, `explosions`, `drops` — each holding `{ mesh, ...state }` objects. `update()` iterates them backwards (`for i = len-1; i >= 0; i--`) so entries can be `splice`d during iteration. Adding a new entity type means: a `THREE.Mesh` added to `scene`, a push into its array, a per-frame block in `update()`, and `scene.remove` + `splice` on death/expiry plus cleanup in `resetGame()` (via `clearArray`). The `explosions` array doubles as a generic **visual decay pool**: `spawnGroundPuff` (zombie landing dust, kill splashes) and the explosion core/shockwave push `{ mesh, life, scale }` entries with per-entity materials that the loop scales, fades, disposes, and splices; only real blasts carry a `light`.
- **Shared resources:** `GEO` and `MAT` cache geometries/materials reused across entities to cut GC. Zombies clone their material per-instance (per-zombie `offsetHSL` grime jitter, safe disposal) and must `material.dispose()` on death — see `killZombie` / `clearArray`. `clearArray` deliberately skips disposing shared `MAT.*` materials.
- **Difficulty:** `wave` object scales spawn interval, `maxZombies`, and per-zombie hp/speed as `wave.time` accumulates (`wave.num` advances every 25 s; `spawnInterval` floors at 1.2 s, `maxZombies` caps at 28). Spawning is driven by a `dt` accumulator in `update()` (not `setInterval`) so it respects pause.
- **HUD extras:** `drawRadar()` paints a player-relative minimap to the `#radar` canvas (throttled to every other frame); high score persists via `localStorage` key `gibsHighScore`.
- **Player state:** the `player` object (hp/ammo/mags/grenades/isReloading/lastShot/speed/radius). The camera *is* the player — there is no separate player mesh; `camera.position` is the player position and `gunMesh` is parented to the camera.
- **State flags:** `isPaused` (toggled by `pointerlockchange`) and `isGameOver` gate input handlers and `update()`.

## Key mechanics and conventions

- **Collisions are distance checks, not physics.** Zombie-hit, item pickup, and grenade blast use `distanceTo` against hardcoded radii (e.g. player `radius` 5 + zombie 3.5). **Bullet-vs-zombie is a swept-segment check** (`segPointDist2`: point-to-segment distance², previous → current bullet position) so shots never tunnel at point-blank or low fps; see the weapons bullet below. Monas collision is an axis-aligned bounding-box test that reverts to the pre-move position; the half-extent differs per entity (player 25, zombie 24, bullet 22 plus `y < 35`). Trees (`treeColliders`, r = 3.4×scale) and the fountain basin (`FOUNTAIN` const) are solid cylinders resolved by `resolveObstacles(pos, radius, feetY)` — horizontal push-out for both player and zombies; the fountain wall is skipped once feet are above its lip, and it returns `true` on a fountain block (the zombie vault trigger — camping on the basin is not safe). Bullets/grenades pass through trees by design.
- **Jump (Space):** `player.vy`/`player.onGround` with `JUMP_V` 33, `GRAVITY` 70 (real seconds, peak ≈ 7.8 units — keep below the 12.5 fence wall); camera y = feet + `EYE_H`. `groundHeightAt(x, z, feetY)` returns the fountain top (`FOUNTAIN.topY`) when landing from above, else 0 — so the fountain is climbable. The fence and Monas remain un-jumpable *for free*: the fence clamp and Monas AABB act on x/z regardless of height. Zombie contact damage stays a 2D `hypot` check, so jumping does not dodge zombies. Space still restarts on game over; reset restores `vy`/`onGround`.
- **Arena = Monas park (`PARK` config).** The playfield is a rectangle `[-PARK.hx..PARK.hx] × [-PARK.hz..PARK.hz]` (currently 620×340 half-extents) built by `createGround`/`createParkRoads`/`createFence`/`createParkProps`: grass interior, central plaza + Monas, diagonal "Jalan Silang" roads, a perimeter **concrete fence** (`FENCE_H`; panel walls + instanced pillars/caps), then the "Jalan Medan Merdeka" ring road (`ROAD_W`), with `createCity` as the distant backdrop. The fence is the player's hard wall — movement is clamped to `PARK.h* - radius - 2`. Trees and the fountain basin are solid (see collisions bullet); the pool and fountain jets stay cosmetic. Trees spawn **only on grass**: `isOnGrass` rejection-sampling excludes the plaza square, both diagonal road strips, the fountain circle, and the pool box. Bump fog/sky/camera-far together if you resize `PARK`.
- **Coordinates are large-scale** (park ~620×340 half-extents, monas ~25 wide, eye height `EYE_H` 11.4 — calibrated so the barrel tip (eye − 2.5 + 0.1) sits level with the zombie chest ≈9.0 (torso spans 5.6–10.2); every feet↔camera conversion must use `EYE_H`). Sizes/distances assume this scale; keep new geometry proportional.
- **Two weapons:** `currentWeapon` is `'rifle'` (AK47: 30-round mags, 130 ms fire delay, 3 s reload) or `'pistol'` (15-round mags, 260 ms, 2.2 s reload); per-weapon ammo lives in `player.rifle` / `player.pistol` (`{ammo, mags, magSize}`). Bullets are identical (same speed/damage/entity); only SFX (`sfxShoot` vs `sfxPistol`), fire delay, and rigs differ. Keys `1`/`2`/`Q` run a 0.5 s switch animation (`switchAnim` timer in `update()`, weapon models/`muzzlePoint`/muzzle-flash rig swapped at the midpoint via `attachMuzzle`); switching cancels an in-progress reload (CS-style). Mag drops refill **both** weapons' mags.
- **Melee (F):** `meleeT`/`meleeCd` timers in `update()`; at 45% of the swing `doMeleeHit()` insta-kills the nearest zombie within `MELEE_RANGE` 16 in a ~±70° front cone (`spawnDrop` + `killZombie`, same as a bullet kill). Swing animation is weapon-specific in `updateDecor` (rifle: sideways stock bash via rotation.y; pistol: downward whip via rotation.x) plus a forward thrust on position.z. Blocked during reload/switch; blocks shooting while swinging.
- **Fire rate / reload use real time:** `Date.now() - player.lastShot > fireDelay` between shots; reload is a per-weapon `setTimeout` whose id is kept in `player.reloadTimer` and **cancelled in `resetGame()` and on weapon switch**. Auto-reload triggers when `ammo` hits 0 with mags left. Bullets spawn at `muzzlePoint` (explicit `Object3D` at the active weapon's barrel tip — never index children) and are stretched into tracers (visual). The hit test sweeps the segment from the bullet's previous position (`b.px/py/pz`, updated in the movement loop) to its current one; **on the spawn frame the segment starts at the player's eye**, because the muzzle sits ~13 units ahead of the camera — without the eye→muzzle segment a zombie standing inside claw range is unshootable (the bullet is born behind it). **Headshot = instant kill:** each zombie is tested head-first against a sphere at `mesh.y + HEAD_Y` (11.5) radius `HEAD_R` (2.4), then the chest sphere (`mesh.y + 6`, r 6); a head hit sets `hp = 0`. Recoil and muzzle flash are visual-only (`gunRecoil` / `muzzleFlash.intensity` decay by `dt`) and deliberately kept off the camera quaternion so they never corrupt mouse-look; the pistol slide kicks with recoil.
- **Input safety:** pointer lock is requested via `requestLock()` (swallows Chrome's NotAllowedError cooldown rejection); `releaseInputs()` clears held keys/mouse/aim on unlock, window blur, and reset. `R` and `G` (grenade) are gated by `!isPaused && !isGameOver`.
- **ADS (right-click = toggle):** `isAiming` drives a smoothed `aimT` (0..1) in `updateDecor`: FOV lerps `BASE_FOV → BASE_FOV/AIM_ZOOM` (70 → ~58, 20% zoom, `updateProjectionMatrix` on change), the gun lerps from hip (3, −2.5, −6) to centered (0, −0.78, −5.2) so the rear-sight notch + front post line up with screen centre, bob is damped, and the crosshair fades out. Purely visual — bullets/grenades still derive from the camera's centre ray, so accuracy/mechanics are unchanged. `releaseInputs()` clears the toggle on pause/blur/reset.
- **Hands + reload rigs:** CS-style first-person arms are children of each weapon group (so they inherit bob/ADS/recoil), built from tactical-glove boxes (palm + individual fingers + thumb + trigger finger) and sleeve cuffs/forearms. `placeLimb(mesh, ax..bz)` stretches a unit-z box between two points (midpoint + scale.z + quaternion) — right forearms are placed once (wrist → off-screen bottom-right elbow); **left forearms are re-placed every frame** in `updateDecor` between a fixed elbow anchor and the moving left hand, so arms stay connected through animations. Each weapon has its own reload keyframe track played over **real time** `Date.now() − reloadStartTime` (in sync with its reload `setTimeout` even across pause): rifle `RELOAD_KF` (3.0 s — `leftHand` from `LEFT_HAND_REST`: grab mag → drop below screen → new mag up, `gunMagMesh` rides the hand t 0.35–2.05 → seat → pull `boltHandle` → return) and pistol `PISTOL_KF` (2.2 s — `pLeftHand` from `P_LEFT_REST`: mag out of the grip, `pistolMagMesh` rides t 0.28–1.5 → insert → rack `pistolSlide` → return). Weapon tilt during reload writes `gunRotX`/`gunRotZ`, which are composed with switch-lower and melee rotations at the end of `updateDecor` — never write `grp.rotation` directly elsewhere; `resetGame` zeroes both groups' rotations.
- **Spawning:** `dt` accumulator in `update()` fires `spawnZombie` every `wave.spawnInterval`s; capped at `wave.maxZombies`. Zombies spawn **outside the fence** (mostly on the side nearest the player) and enter via a two-phase state machine: `state:'jumping'` arcs them over the fence (sine parabola, lerping `s*→l*`), then `state:'chasing'` runs the normal ground AI. The jump arc is generalized — `y = baseY + lerp(jumpY0, jumpY1, t) + sin(πt)·arcH` — and is **reused as a fountain vault**: a chasing zombie blocked by the basin wall (`resolveObstacles` returns true) re-enters `'jumping'` onto the basin top when `segmentHitsFountain` confirms the path to the player crosses it (`vaultCd` 1.5s prevents ping-pong loops). `z.groundY` tracks the current floor (0 or `FOUNTAIN.topY`) and snaps to 0 when they walk off the rim. **Zombies attack by clawing, not kamikaze:** a chasing zombie stops advancing inside claw reach (`player.radius + 3.0`) and, when `z.attackCd` ≤ 0 and within `player.radius + 3.5`, deals 1 damage, sets `attackCd = CLAW_CD` (3 s per zombie), and plays a one-arm swipe (`clawT`/`clawSide`, animated in `animateZombieRig` over the current pose). While standing in claw range (`z.moving === false`) the walk cycle stops entirely — legs/bob damp to rest and only a small breathing sway remains. The zombie persists and keeps clawing. Jumping/vaulting zombies are shootable but can't claw until `chasing`. The bullet hit-test offsets both spheres from `mesh.y` (chest `+ (isModel ? 6 : 0)`, head `+ HEAD_Y`) so it works mid-air, on grounded models (feet at y=0), and on the basin top.
- **SFX:** always play via `playSFX(sfx, vol = 0.7)`, which round-robins a small per-clip pool (max 8 cloned `Audio` nodes) so overlapping sounds work without per-shot allocation. Clips in `assets/sounds/`: `gun-shoot`, `pistol-shoot`, `grenade-explode`, `reload` (shared by both weapons), `pick-up-item` (drop pickup), `smash-melee-attack` (melee swing, plays even on a whiff), `throwing-grenade`, `player-footstep` (walk/sprint cadence via `footT` timer + landing after a jump, at reduced volume), `zombie-attack-melee` (claw hit — plays together with `jokowi-kaget`, the player-damage scream `sfxHit`), `zombie-step` (**one global source, not per-zombie**: every ~0.55 s the `zStepT` timer plays a single step for the nearest moving chasing zombie within 120 units, volume scaled by distance — keep it this way for performance). `zombie-spawn.mp3` exists on disk but is **no longer played**.
- **Zombie model = procedural blocky human** (story: civilians turned by aliens). `buildHumanZombie()` assembles one per spawn from shared `ZG` geometries (~13 units tall, feet at y=0, facing +Z so the AI's `lookAt` works directly): torso, multi-material head (shared canvas face texture `zombieFaceTex` on the +Z face, index 4, tinted by skin tone), hip/knee/shoulder pivot groups, and profession accessories. Six `ZOMBIE_VARIANTS`: warga (random shirt colors), pekerja (orange vest + stripe + yellow helmet), polisi (cap + brim + gold badge), pedagang (apron + caping cone), penjaga toko (blue vest + name tag), kantoran (tie). Skin from `ZOMBIE_SKIN_TONES`; every material is per-instance (`offsetHSL` grime jitter); `disposeZombie` handles both single and **array** materials (the head) — never dispose `ZG` geometries or `zombieFaceTex`. **Zombies show no damage feedback by design** (no hit tint, no low-hp color — the old `tintZombie` was removed on purpose; the player must not be able to tell how hurt a zombie is). `isModel` stays `true` on every zombie so hit constants (chest +6, hitR 6, head +11.5 r 2.4, baseY 0) are untouched.
- **Zombie animation:** `animateZombieRig` rotates the pivot groups' `rotation.x` directly (lateral axis = local X because the rig faces +Z and the group is `lookAt`-ed at the player) — no quaternion math, no `AnimationMixer`. Walk = alternating thigh/shin/arm swing + body bob (`inner.position.y`) + head sway; base arm pose is the classic zombie forward reach (−1.15 rad); jump/vault = static tuck. Amplitudes are the `SW`/`KN`/`AR` constants.
- **`resetGame()`** must clear every entity array and `scene.remove` its meshes — keep it in sync when adding entity types.

## Editing guidance (from AGENTS.md)

- Keep changes compatible with a single static page. Do not introduce build tools, frameworks, or package dependencies unless the user explicitly asks to modernize the project structure.
- Only split inline script/CSS into separate files if the user requests it.
- Preserve existing controls (WASD move, Shift sprint, mouse look, left-click fire, right-click toggle aim-down-sights, 1/2/Q switch weapon, F melee, G grenade, R reload, Space to jump — or restart on the game-over screen) and core mechanics unless the user asks for gameplay changes.

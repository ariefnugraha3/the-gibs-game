## Architecture (refactored 2026-07-05 into ES modules)

The old single-file `index.html` was split into **ES modules under `src/`** with a **scene system**. There is still **no build step, no npm dependencies, no framework** вЂ” plain static files; Three.js r128 + post-processing examples come from CDN `<script>` tags as the global `THREE` (modules do NOT import it). `package.json` is metadata-only (`"type": "module"` for Node tooling).

**в†’ The complete module catalog, every export, the scene interface contract, the config key table, and the how-to-add-a-stage recipe live in [MODULES.md](MODULES.md). Read it before hunting through files.**

Load-bearing rules (details in MODULES.md):

- **Scenes:** `menu` (DOM-only) в†’ `survivalScene` or `campaign stage1Scene в†’ stage2Scene`, managed by `core/sceneManager.js` (`setScene`/`activeScene`). All mode-specific behavior (collision walls, ground height, bullet blockers, grenade bounds, robot movement AI, radar landmarks, HUD status, drop clamping, win check, restart policy) is behind `activeScene.*` hooks вЂ” **shared systems contain zero mode if-else**. Adding a campaign stage = one new file in `src/scenes/campaign/` + wiring per the MODULES.md recipe.
- **The campaign stage worlds coexist in one THREE.Scene** far apart (all THREE stages are now indoor buildings: stage-1 at xв‰€30000, stage-2 at xв‰€60000, stage-3 at xв‰€90000 вЂ” no more shared survival park); `camera.far` 4000 + robot distance culling hide the far ones. Stage 1 & Stage 2 buildings are built ONCE at campaign start by `stage1.enter` (`buildStage2World()` then `buildWorld()`); the **Stage 3 building is built lazily (guarded) in `stage3.enter`** on the stage-2в†’3 transition (stage 3 is entered once per playthrough; restart always goes to stage 1). `placeStage2Robots()` (stage 2's `placeRobots`, which also places stage-2 supplies) + stage-1 `placeRobots`/`placeSupplies` run every stage-1 enter; stage-3 robots+supplies are placed in `stage3.enter`. Campaign restart (death) always re-enters stage 1 (arrays cleared by `resetGame` first). Every campaign robot carries `z.stage`; stage transition disposes the previous stage's robots silently.
- **All gameplay-tunable numbers live in `config/gameplay.json`**, loaded into `CFG` (`core/config.js`) before the game starts. Read `CFG.x.y` inside functions only (never at module top level). New tuning values go in the JSON, not as code constants. Visual-only values (FOV, animation amplitudes, colors) intentionally stay in code. (The reload keyframe auto-scaling note is historical вЂ” reload was removed with the magazine system 2026-07-11.)
- **Cross-module state = live ESM bindings**: the owning module exports `let` + setters (`isPaused`/`score` in `core/state.js`, `stamina`/`dodgeActive`/`dodgeProgress` in `entities/player.js`, `currentWeapon`/`isAiming`/`switchAnim` in `entities/weapons.js`, `scene`/`camera`/`renderer` in `core/renderer.js`). Circular imports are fine and used (weaponsв†”player, robotsв†”effects, stage1в†”stage2, gameв†”input) **as long as bindings are only used inside functions**.
- **`updateGame()` block order in `core/game.js` is a contract** (matches the old `update()`): scene updateMode в†’ weapon timers в†’ player movement в†’ weapon state в†’ shooting в†’ grenades в†’ explosions в†’ blood в†’ drops в†’ bullets в†’ robots в†’ win check. Bullets must move before the robot sweep hit-test.
- **The old `camera` is now a LOGIC PIVOT, not the renderer (top-down pivot 2026-07-11).** Its position = the player's eye-height point (feet + `eyeHCur`, same semantics as the FPS era) and its yaw = the AIM direction toward the cursor (set every frame by `updateTopdownAim` in input.js). Every gameplay system (robot targeting/claws, bullet/grenade/melee direction via `camera.getWorldDirection`, pickups, scene collision, radar, spawn points set by scenes) still reads `camera` and is UNCHANGED. Rendering is done by **`viewCam`** (renderer.js), a separate camera at fixed offset `CAM_OFF` (0, 116, 100 вЂ” brought closer in steps 2026-07-11 [185/100 в†’ 161/87 в†’ 146/79 в†’ 130/82]; then 130/82 в†’ 116/100 LOWERED the angle a further ~15% while KEEPING the distance ~154 units [y down, z up] so only the pitch flattens, not the zoom: ~57.8В° в†’ ~49.2В°) вЂ” Alien Shooter oblique angle, fixed yaw facing в€’z, so **screen-up = world в€’z**. `followViewCam()` tracks a **dead-zone focus point** `camFocus` (not the pivot directly): it only shifts when the pivot leaves a tolerance box of world half-extents `DEAD_X`/`DEAD_Z` (`DEAD_Z` for north/south is intentionally smaller so the widescreen forward view isn't cramped), so small movement doesn't move the camera (anti-dizzy, per user); y follows without tolerance; jumps > 400 units (spawn/scene-change/restart) snap-recenter. ## Key mechanics and conventions

- **Collisions are distance checks, not physics.** Robot-hit, pickup, and blast use `distanceTo` radii. **Bullet-vs-robot is a swept-segment check** (`segPointDist2`, prevв†’current position; spawn-frame segment starts at the player's EYE so point-blank never tunnels). Survival: Monas is a solid AABB resolved PER-AXIS (`resolveMonas` in world.js вЂ” `MONAS_HALF` 22, base 44Г—44; hug-and-slide, NOT a full revert, so the player never sticks вЂ” mirrors the campaign wall fix), solid trees/fountain cylinders (`resolveObstacles`), fence clamp. **`bulletBlocked` follows the real Monas silhouette height-by-height** (base 44 low в†’ thin obelisk ~16 at eye height, nothing above y 64) so bullets pass through the empty air beside the obelisk and hit robots behind вЂ” do NOT revert it to a flat 44-wide column. Campaign: walkable-union tests (`stage1Walk` grid / `highwayWalk`) with per-axis `slideWalk` (hug walls, don't stick) + rotated-AABB `resolveBlockers` (medians/wrecks/furniture; standable tops are walkable). Robot bodies are solid to the player only (push-out at `CFG.robot.bodyBlockRadius` 7.5, deliberately < claw-stop 8.0 so an attacking robot doesn't shove the player every frame).
- **Frame-rate independence:** `dt` (clamped 0.05) and `step = dt*60`. Multiply motion by `step`; decrement timers by `dt`. Fire-rate/reload use real `Date.now()` time.
- **Entity model:** six arrays in `core/state.js` вЂ” `bullets, enemyBullets, robots, grenades, explosions, drops` вЂ” iterated backwards for splicing (`enemyBullets` = robot ranged fire в†’ player, added 2026-07-12). `explosions` doubles as a generic visual-decay pool (ground puffs ride it). "Blood" (GREEN COOLANT since 2026-07-12 вЂ” robots bleed coolant; `Blood*` function names kept) is a **fixed pool of 72 sprites** (`effects.js`, each with a velocity so it SPRAYS + falls вЂ” `spawnBloodBurst(x,y,z,dirx,dirz,n,power,spread)`; `spread` = cone width, `6.283` = 360В° for explosions), explosion flashes a **fixed pool of 3 lights** вЂ” keep point-light counts constant (no mid-game shader recompiles). New entity type = mesh + array + update block + `resetGame` cleanup.
- **GORE (`entities/gore.js`, 2026-07-11; re-themed GREEN COOLANT 2026-07-12):** robots don't vanish on death вЂ” `killRobot(i, {cause,dirx,dirz})` **splices the robot from the live `robots` array FIRST** (so the wreck is inert: no AI/claw/body-block/bullet-hit and it doesn't hold up the survival wave-clear check), awards score, then **hands the robot's own mesh to `spawnCorpse`** (a topple animation about the feet в†’ brief lie в†’ fade [materials flipped `transparent`, no recompile] + slight sink в†’ dispose ~1.7 s) and flings **gibs** (`gibRobot` hides limb pivots on the wreck and throws chunks colored per that limb's armor plate вЂ” machine scrap, default tone `0x3d444c`). Robots "bleed" **GREEN COOLANT**, the player bleeds **RED** (2026-07-12): the splat sprite texture (effects.js) is NEUTRAL white and tinted per-spawn via `material.color` вЂ” default `COOLANT_HEX` green, player-damage sites pass `PLAYER_BLOOD_HEX` red through the trailing `color` param of `spawnBlood`/`spawnBloodBurst`; the ground `spawnBloodDecal` puddle texture/tint (gore.js, default `0x2fbf66`) stays green (robot-only) вЂ” the function names kept their `Blood*` names, only the theme changed. **`cause: 'explosion'` = extra-gory dismemberment** (2Г— omnidirectional [360В°] coolant bursts ~52 sprites, the 5 limb gibs PLUS ~10 extra metal-scrap chunks, and ~9 coolant puddles SCATTERED around the blast + one per gib landing); **`cause: 'melee'` (player sword kill, 2026-07-13) = the robot is CUT IN HALF at the waist**: `bisectCorpse` (gore.js) neutralizes the inner pose then reparents every `inner` child at local y в‰Ґ 6.3 (torso/chest/core/pack/shoulders/neck/head/both arms) into a NEW top-half group whose origin sits exactly ON the cut plane (local y 6.35 вЂ” scale-independent, works for the boss too) вЂ” the top half is FLUNG tumbling (waist-pivot world-axis quaternion spin) toward aim + blade-travel-left, DRIPPING coolant every 0.06 s until it lands (+decal), while the legs+pelvis (the ORIGINAL group) keep STANDING 0.5вЂ“0.85 s with slowly buckling knees and a GUSHING stump, then topple like a normal corpse; both cross-sections get glowing orange CUT-EDGE discs (shared `CUT_GEO` circle + per-wreck Lambert-emissive clones = the robot-visor shader program, zero recompiles), both halves fade/sink together and the robot's materials (SHARED across the halves вЂ” collected uniquely from both trees) are disposed ONCE (internal `bisected` pool, cleaned by `resetGore`);

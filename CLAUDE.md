# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Overview

Single-page browser game: "Gibran vs Zombie 3D (DOOM FPS)". A first-person Three.js shooter. **All user-facing UI text MUST be in English — permanent user rule (2026-07-05), no exceptions, applies to every future change.** Code comments are in Indonesian. **Two game modes** picked on the `#modeSelect` screen at load (before the cutscene): **Survival** (defend Monas against endless zombie waves — the original mode) and **Campaign**, a two-stage linear mode: **Stage 1** "Gedung Terbengkalai (Lantai 3)" — an indoor 60×60 m office-building floor (grid-based) — then via the exit stairwell **Stage 2**, the ruined-Jakarta highway map. Campaign zombies lurk motionless until seen/approached.

> **Desktop/Steam port:** a full plan to wrap this into a Windows `.exe` (Electron) and ship on Steam/itch lives in [STEAM-DESKTOP-PLAN.md](STEAM-DESKTOP-PLAN.md) — consult it if the user asks about desktop builds, `.exe`, or Steam publishing.

## Running

No build system, package manager, or tests. Open `index.html` directly in a browser, or serve the folder with a static HTTP server (preferred — required for the `.mp3` SFX to load reliably and for PointerLock):

```
python -m http.server 8000      # then open http://localhost:8000
```

Three.js r128 and the post-processing example scripts are loaded from CDNs (`<head>` scripts), so an internet connection is required.

## Architecture

Everything lives in [index.html](index.html) — inline CSS, DOM overlay, and the full game in one `<script>`. There are no modules; all functions and state are top-level globals. The only assets are SFX in `assets/sounds/` (`*.mp3`); all visuals (textures, zombies, weapons) are procedural. `assets/visuals/zombie.glb` still exists on disk but is **no longer used**.

- **Entry point / mode gate:** `init()` builds the scene then `animate()` starts the render loop. **They are no longer called at script load** — the `initModeSelect()` IIFE waits for a click on a `#modeSelect` card (`data-mode` `survival`/`campaign`), sets the `GAME_MODE` global, hides the overlay, then calls `init()`+`animate()`. `GAME_MODE` (checked as `isCamp = GAME_MODE === 'campaign'` at the top of `update()`) branches world-building in `init()` and every mode-specific block in the game loop. `#modeSelect` (z-index 30) sits above the cutscene (z 20) and blocker (z 5).
- **Opening cutscene:** a self-invoking `initCutscene()` IIFE (bottom of the script) runs a pure-DOM/CSS 4-slide slideshow (`#cutscene` overlay). It's the Monas backstory, so it only shows for **Survival** — the campaign card hides `#cutscene` immediately (goes straight to `#blocker`). It's independent of the Three.js loop — `finish()` just hides the overlay to reveal `#blocker`, whose click requests PointerLock. No game state is touched here.
- **Campaign stages:** both stage worlds are built once at `init()` and coexist in one scene, separated by distance (the building sits at x≈30000, ~26 km from the highway; `camera.far` 4000 clips the far one, zombie culling hides its zombies). `CAMP_STAGE` (1|2) is the single switch: `inCampWalkable(x,z,r)` **dispatches** to `stage1Walk` (building grid) or `highwayWalk` (road union) — placement code must call the specific fn directly (`placeCampaignZombies` uses `highwayWalk`) because placement runs while `CAMP_STAGE` is still 1. `campResolve`/`campGroundHeight` need no dispatch (blockers carry distance prechecks; fountains just never match far positions). Reaching stage 1's exit-stairs trigger rect (`S1_EXIT`) calls `enterStage2()`: silently disposes remaining stage-1 zombies (no score/drops), teleports to `CAMP_START`, swaps fog/light presets via `setStageLighting(stage)` (uniform-only — no shader recompile) and shows `#stageMsg`. Death always restarts at stage 1 (`resetGame` re-places both zombie sets + supplies). Win = stage 2 cleared (`CAMP_STAGE===2 && zombies.length===0` → "MISI SELESAI"); every zombie has a `z.stage` tag and the HUD counts only the active stage's.
- **Campaign stage 1 — "Gedung Terbengkalai (Lantai 3)" (`createStage1World` + `S1` config):** indoor 60×60 m office floor from the user's floor-plan image: start stairwell (NW) → lobby → Kantor 1/2 → Ruang Persediaan (NE, supply pickups) → central corridor → toilet/utilitas → area pemuatan → ruang pendingin → exit stairwell (SE, stage-2 trigger). Built from a **30×30 grid of 2 m cells** (`buildS1Grid`: all-wall then `carve()` rooms + 2-cell doorways; `s1grid` 1=wall 0=floor). The grid is the single source of truth: walls render as one `InstancedMesh` (only cells adjacent to floor), `stage1Walk` is the collision test, `s1LOS` (cell-sampled line) gates zombie activation, and `s1SegHitsWall` kills **bullets on walls** (no shooting dormant zombies through rooms). Ceiling plane at `S1.H` 22 (grenades bounce off it in stage 1); furniture (desks/shelves/crates/freezers, one instanced mesh) doubles as `campBlockers` (desk-height ones standable/climbable). Interior lighting: dimmed ambient/hemi/dir preset + **6 fixed PointLights** created at init (before first render — one-time shader compile; only intensities animate: `s1FlickerLight` corridor flicker in `updateDecor`) + emissive lamp fixtures and a green EXIT sign. If you edit the room layout, re-verify connectivity (BFS from start over floor cells) and that no furniture blocker parks in front of a doorway.
- **Campaign zombies:** all spawn `state:'idle'`, `hp:3` (3 body shots / 1 headshot), tagged `z.stage`, speed `(0.6..1.0)×0.5` (≈2.6–4.3 m/s — deliberately faster than survival's ×0.3 baseline). **Stage 1**: 11 fixed spawn cells (`S1_ZOMBIES`, skull marks from the plan) with jitter, snapped out of furniture. They wake when the player is within `CAMP_ACTIVATE` (15 m) **and** has grid line-of-sight (`s1LOS`) — or is <30 u through a thin wall — or the zombie is shot, or an explosion lands within 250. **Stage 2 (highway, `placeCampaignZombies`)**: ~52 rejection-sampled along the roads (min 42-u spacing; the bottom-fountain area within `START_CLEAR` stays empty so the highway entrance is calm); pure distance activation (no LOS). Chasing campaign zombies use `campResolve` + per-axis walkable slide (they hug walls instead of sticking). Beyond `CAMP_VIS` (1500) idle zombies are `visible=false` and skip AI/hit loops — this is also what hides the *other* stage's zombies. Idle zombies are solid to the player (body-push includes `'idle'`). `GAME_MODE`/`CAMP_STAGE` also branch `updateUI`, `drawRadar` (stage 1: green exit marker; stage 2: fountains+wrecks; survival: Monas), `spawnDrop` (no PARK clamp), and grenade/player collision.
- **Supplies & medkit:** `placeStage1Supplies()` stocks the Ruang Persediaan (4 mag + 2 grenades + 3 medkits total, incl. one in the toilet and a mag in the loading bay) as non-expiring `drops` (`timer:1e9`) — re-placed by `resetGame` since `clearArray(drops)` wipes them. **`'medkit'` is a third drop type** (`buildMedkitMesh`, shared `MEDKIT_MAT` — Groups aren't traversed by `clearArray`, so shared materials must be used): pickup heals +2 (cap `PLAYER_MAX_HP`); radar plots it pink. Medkits are never random zombie drops. **Full-item rule:** any drop the player touches while already at cap (mags at `MAX_MAGS` on both weapons, grenades at `MAX_NADES`, HP at max) is *not* consumed — it stays on the ground and the pickup feed shows a grey "... already full" line, rate-limited by `fullInfoCd` 1.2 s so standing on the item doesn't spam.
- **Environment is decorative:** `createSky` (canvas-gradient sky dome with stars/smoke + moon and additive halo — all now children of `skyDome`, recentred on the player each frame), `createCity` (survival only: two building rings at radius ~820–1380, outside the park + ring road, with rubble piles at building feet), and `createEmbers` (a `THREE.Points` ash field, both modes) build the apocalyptic backdrop. None of it has collision. Ambient animation lives in `updateEmbers` (upward drift + wind sway, recentered on the player) and `updateDecor` (runs every frame even while paused: shadow camera follows the player, Monas flame + burning-building flicker, fountain jets pulse, muzzle-flash sprite fade).
- **Rendering pipeline:** sRGB output + ACES filmic tone mapping (`toneMappingExposure` is the single brightness knob) + PCFSoft shadow maps (2048px, ortho ±320 around the player; `dirLight.target` is in the scene so `updateDecor` can move both). Casters: zombies, trees, fence, Monas, fence rails; the distant city does not cast. **Keep the point-light count constant** — explosion flashes reuse the 3-light `explosionLights` pool (intensity 0 when idle) so Three.js never recompiles shaders mid-game; `clearArray`/the explosion loop set `o.light.intensity = 0` instead of removing lights.
- **Post-processing:** `composer` chain RenderPass → UnrealBloomPass (0.45/0.4/0.72, kept in `bloomPass`) → GammaCorrectionShader (the RTs are linear, so this pass does the sRGB conversion) → FXAAShader (`setFxaaRes()` must be called on resize). Eight extra CDN example scripts feed it; if any fail to load, `composer` stays `null` and `animate()` falls back to plain `renderer.render`. Materials that must bloom hard set `toneMapped: false` (tracers, explosion core/shock, flame glow, muzzle sprite, fire sprites).
- **Graphics quality is a player setting on the start screen:** `QUALITY` tiers 0–4 (Penuh/Tinggi/Sedang/Rendah/Sangat Rendah) + `applyQuality(t)`: pixel ratio (capped dpr 2 → 1.5 → 1 → 0.8 → 0.6), shadow-map size (2048 → 1024 → 512 → **0 = `dirLight.castShadow` off**), `bloomPass.enabled`, and `postFxOn` (tiers 3–4 skip the composer entirely — direct render, where canvas MSAA covers AA). Most knobs avoid shader recompiles; the tier-4 `castShadow` toggle *does* recompile once, which is fine **only because** quality is locked to the start screen — don't call `applyQuality` mid-game. The `QUALITY` const must stay in the top globals block, declared **before** the `init()` call (`initQualityUI` reads it during init; a later `const` is TDZ and kills the whole script — this bug happened once). `initQualityUI()` wires the `#qualityRow` buttons inside `#blocker` (clicks `stopPropagation` so they don't trigger the blocker's start-click); the choice persists in `localStorage` key `gibsQuality`, defaulting to tier 1 on weak devices (`deviceMemory ≤ 4` or `hardwareConcurrency ≤ 4`) else tier 0. The row is hidden permanently on the first pointer lock — quality is selectable **only before the first start**, not on pause. Shadow-caster budget: zombie small parts (feet/hands/accessories) and grenade detail parts don't cast — keep `castShadow` off tiny meshes.
- **Instancing:** city buildings (normal / burning / rubble), park trees (trunk + 2 canopy layers), and fence posts are `THREE.InstancedMesh` with per-instance colors — a handful of draw calls instead of ~500. All instanced meshes set `frustumCulled = false` because r128 doesn't compute merged instance bounds (otherwise they vanish when the unit geometry leaves the frustum).
- **Textures are procedural:** `makeTexture(w, h, drawFn, repX, repY)` + `speckle()` generate every color map (grass blades, asphalt + cracks + lane dashes, radial plaza paving, marble, wood grain, building facades/lit windows, water ripples, muzzle star-flash, moon craters, fire/halo/ember dots) on canvas — no image files. `makeNormalMap(w, h, heightDrawFn, strength)` (+ `noiseHeight`) bakes normal maps from canvas heightmaps; normal maps stay linear (no sRGB), and in r128 the UV transform comes from `material.map`, so a shared normal map follows each material's diffuse repeat. Canvas color textures are flagged `sRGBEncoding`.
- **Game loop:** `animate()` → `requestAnimationFrame` → `update(dt, step)` → `renderer.render`. `update()` early-returns when `isPaused` or `isGameOver`, so all gameplay (movement, shooting, AI, collisions) only advances while pointer-locked and alive.
- **Frame-rate independence:** `animate()` derives `dt` from a `THREE.Clock` (clamped to 0.05 to absorb tab-switch spikes) and `step = dt * 60`. Speed constants are calibrated at 60 fps and multiplied by `step`; per-frame timers (drops, explosions, waves) decrement by `dt` in seconds. When adding motion, multiply by `step`; when adding a countdown, subtract `dt`.
- **Entity model:** five arrays — `bullets`, `zombies`, `grenades`, `explosions`, `drops` — each holding `{ mesh, ...state }` objects. `update()` iterates them backwards (`for i = len-1; i >= 0; i--`) so entries can be `splice`d during iteration. Adding a new entity type means: a `THREE.Mesh` added to `scene`, a push into its array, a per-frame block in `update()`, and `scene.remove` + `splice` on death/expiry plus cleanup in `resetGame()` (via `clearArray`). The `explosions` array doubles as a generic **visual decay pool**: `spawnGroundPuff` (zombie landing dust, kill splashes) and the explosion core/shockwave push `{ mesh, life, scale }` entries with per-entity materials that the loop scales, fades, disposes, and splices; only real blasts carry a `light`. Bullet-impact blood is different: `bloodPool` is a **fixed pool of 14 sprites** (one shared procedural splat texture, built once in `init`, `visible = false` when idle) reused round-robin by `spawnBlood(x, y, z)` — zero per-hit allocation/dispose by design, keep it that way; the sprite is nudged ~1.5 units toward the camera so it isn't buried inside the zombie, faded in `update()`, and hidden (not removed) by `resetGame`.
- **Shared resources:** `GEO` and `MAT` cache geometries/materials reused across entities to cut GC. Zombies clone their material per-instance (per-zombie `offsetHSL` grime jitter, safe disposal) and must `material.dispose()` on death — see `killZombie` / `clearArray`. `clearArray` deliberately skips disposing shared `MAT.*` materials.
- **Difficulty:** `wave` object scales spawn interval, `maxZombies`, and per-zombie hp/speed as `wave.time` accumulates (`wave.num` advances every 25 s; `spawnInterval` floors at 1.2 s, `maxZombies` caps at 28). Spawning is driven by a `dt` accumulator in `update()` (not `setInterval`) so it respects pause.
- **HUD extras:** `drawRadar()` paints a player-relative minimap to the `#radar` canvas (throttled to every other frame); high score persists via `localStorage` key `gibsHighScore`. **The `#crosshair` element is permanently hidden via CSS `display:none` (user request)** — the element and every JS write to it (ADS opacity fade, `.hit` flash) are intact so shooting/recoil mechanics stay byte-identical; don't "clean them up" and don't re-show it. Item pickups push an English line into `#pickupFeed` (left side of screen) via `showPickup(text, color)` — appear → hold 2 s → fade, max 5 entries.
- **Campaign wall sliding:** `campSlideWalk(pos, oldX, oldZ, r)` replaces the old revert-both-axes clamp for the player (called twice: after movement and again after `campResolve`): if the new position is not walkable it keeps whichever axis component is still valid, so hitting a wall at an angle glides along it instead of sticking (the "stuck on wall" bug). Chasing zombies use the same per-axis pattern inline.
- **Player state:** the `player` object (hp/ammo/mags/grenades/isReloading/lastShot/speed/radius). Max HP is `PLAYER_MAX_HP` 10 (claw damage stays 1); grenades cap at `MAX_NADES` 5, reserve mags at `MAX_MAGS` 5 per weapon. The camera *is* the player — there is no separate player mesh; `camera.position` is the player position and `gunMesh` is parented to the camera.
- **State flags:** `isPaused` (toggled by `pointerlockchange`) and `isGameOver` gate input handlers and `update()`.

## Key mechanics and conventions

- **Collisions are distance checks, not physics.** Zombie-hit, item pickup, and grenade blast use `distanceTo` against hardcoded radii (e.g. player `radius` 5 + zombie 3.5). **Bullet-vs-zombie is a swept-segment check** (`segPointDist2`: point-to-segment distance², previous → current bullet position) so shots never tunnel at point-blank or low fps; see the weapons bullet below. Monas collision is an axis-aligned bounding-box test that reverts to the pre-move position; the half-extent differs per entity (player 25, zombie 24, bullet 22 plus `y < 35`). Trees (`treeColliders`, r = 3.4×scale) and the fountain basin (`FOUNTAIN` const) are solid cylinders resolved by `resolveObstacles(pos, radius, feetY)` — horizontal push-out for both player and zombies; the fountain wall is skipped once feet are above its lip, and it returns `true` on a fountain block (the zombie vault trigger — camping on the basin is not safe). Bullets pass through trees by design; grenades do **not** — they run `resolveObstacles` so their rolling looks right. Zombie bodies are solid **for the player only**: a 2D push-out at `ZBODY_R` 7.5 in the movement block shoves the player out of chasing zombies (jumping/vaulting ones are skipped, zombie movement is untouched); `ZBODY_R` is deliberately smaller than the claw stop distance (`player.radius + 3.0` = 8.0) so a zombie standing in claw range doesn't shove the player every frame — keep that inequality if you tune either constant. The fence clamp / Monas AABB / tree-basin resolve run *after* the push so it can't force the player through walls.
- **Jump (Space):** `player.vy`/`player.onGround` with `JUMP_V` 33, `GRAVITY` 70 (real seconds, peak ≈ 7.8 units — keep below the 12.5 fence wall); camera y = feet + `EYE_H`. `groundHeightAt(x, z, feetY)` returns the fountain top (`FOUNTAIN.topY`) when landing from above, else 0 — so the fountain is climbable. The fence and Monas remain un-jumpable *for free*: the fence clamp and Monas AABB act on x/z regardless of height. Zombie contact damage stays a 2D `hypot` check, so jumping does not dodge zombies. Space still restarts on game over; reset restores `vy`/`onGround`.
- **Arena = Monas park (`PARK` config).** The playfield is a rectangle `[-PARK.hx..PARK.hx] × [-PARK.hz..PARK.hz]` (currently 620×340 half-extents) built by `createGround`/`createParkRoads`/`createFence`/`createParkProps`: grass interior, central plaza + Monas, diagonal "Jalan Silang" roads, a perimeter **concrete fence** (`FENCE_H`; panel walls + instanced pillars/caps), then the "Jalan Medan Merdeka" ring road (`ROAD_W`), with `createCity` as the distant backdrop. The fence is the player's hard wall — movement is clamped to `PARK.h* - radius - 2`. Trees and the fountain basin are solid (see collisions bullet); the pool and fountain jets stay cosmetic. Trees spawn **only on grass**: `isOnGrass` rejection-sampling excludes the plaza square, both diagonal road strips, the fountain circle, and the pool box. Bump fog/sky/camera-far together if you resize `PARK`.
- **Coordinates are large-scale** (park ~620×340 half-extents, monas ~25 wide, standing eye height `EYE_H` 11.4 — calibrated so the barrel tip (eye − 2.5 + 0.1) sits level with the zombie chest ≈9.0 (torso spans 5.6–10.2)). The *effective* eye height is `eyeHCur` (= `EYE_H − CROUCH_DROP·crouchT`, lowered while crouching): every feet↔camera conversion inside `update()` must use `eyeHCur`, and the crouch block deliberately computes `feetY` with the *previous* frame's `eyeHCur` before re-smoothing `crouchT`, so feet stay glued to the ground during the transition (no phantom fall/landing SFX). `EYE_H` itself is only for spawn/reset positions. Sizes/distances assume this scale; keep new geometry proportional.
- **Two weapons:** `currentWeapon` is `'rifle'` (**displayed as "Assault Rifle"** — internal ids/comments still say rifle/AK47: 30-round mags, 130 ms fire delay, 3 s reload) or `'pistol'` (15-round mags, 260 ms, 2.2 s reload); per-weapon ammo lives in `player.rifle` / `player.pistol` (`{ammo, mags, magSize}`). Bullets are identical (same speed/damage/entity); only SFX (`sfxShoot` vs `sfxPistol`), fire delay, and rigs differ. Keys `1`/`2`/`Q` run a 0.5 s switch animation (`switchAnim` timer in `update()`, weapon models/`muzzlePoint`/muzzle-flash rig swapped at the midpoint via `attachMuzzle`); switching cancels an in-progress reload (CS-style). Mag drops refill **both** weapons' mags, capped at `MAX_MAGS` 5 each; a mag pickup is skipped (left on the ground) only when both weapons are already at the cap.
- **Grenade (G) is ballistic with a fuse, not on-impact:** `handleThrowGrenade` gives the grenade a velocity along the camera ray (`NADE_V` 110 + `NADE_UP` 18 upward; aiming up throws farther) and it explodes only when `fuse` (`NADE_FUSE` 3 s) runs out — wherever it is, even mid-air. The per-frame loop integrates real-seconds gravity (same `GRAVITY` 70), bounces off the floor (`vy < −9` → reflect ×0.42, horizontal ×0.72) then rolls with friction (2.0/s decay), bounces off the fence bounds and the Monas AABB (velocity reflected ×0.45), collides with trees/basin via `resolveObstacles`, can land on the fountain top via `groundHeightAt`, and spins the mesh around `up × velocity` at `speed/NADE_R` for the rolling visual. Grenade meshes (thrown *and* the drop item, scaled 0.8) come from `buildGrenadeMesh()` — a Group assembled from shared `NADE_GEO`/`NADE_MAT` (Mk2 pineapple body + fuse head + lever + pin ring); never dispose those shared resources (the old `GEO.grenade`/`MAT.grenade`/`*.dropNade` entries are now unused but kept for the `clearArray` guards). On its **first floor contact** each grenade plays `grenade-rolling.mp3` once, gated to ≤90 units from the player with distance-scaled volume (far throws are silent by design). **A landed grenade is solid to the player and zombies:** early in the grenade loop (after integration, *before* the wall/floor resolve so a shove into a wall is corrected the same frame), `nudgeGrenade(g, px, pz, radius)` runs for the player (`player.radius`) and every non-jumping zombie (3.5) against grenades that have `rolled` and sit low (`y < NADE_R + 5`). It positionally ejects the grenade to the pusher's edge (no penetration) and raises its along-normal speed **up to a cap `NADE_PUSH_SPEED` 22** (never above — so sprinting into one doesn't punt it; a grenade already faster in that direction is left alone). The cap + existing 2.0/s friction give a natural ~1.5 m roll-out, deliberately **not** a kicked-ball launch. Only the thrown `grenades` array is solid — the `drops` pickup items are still walked over to collect.
- **Melee (F):** costs `STA_MELEE` 15 stamina per swing (keydown ignored below 15). `meleeT`/`meleeCd` timers in `update()`; at 45% of the swing `doMeleeHit()` insta-kills the nearest zombie within `MELEE_RANGE` 16 in a ~±70° front cone (`spawnDrop` + `killZombie`, same as a bullet kill). Swing animation is weapon-specific in `updateDecor` (rifle: sideways stock bash via rotation.y; pistol: downward whip via rotation.x) plus a forward thrust on position.z. Blocked during reload/switch; blocks shooting while swinging.
- **Fire rate / reload use real time:** `Date.now() - player.lastShot > fireDelay` between shots; reload is a per-weapon `setTimeout` whose id is kept in `player.reloadTimer` and **cancelled in `resetGame()` and on weapon switch**. Auto-reload triggers when `ammo` hits 0 with mags left. Bullets spawn at `muzzlePoint` (explicit `Object3D` at the active weapon's barrel tip — never index children) with `speed` 27 (×`step`, `life` 120) and are stretched into tracers (visual, scale.z 8.5 — scaled with speed so the fast round reads as a continuous streak, not spaced dots). Because the hit test is a swept segment (prev→current), raising the speed is safe — no tunneling even at point-blank / low fps; `life` decrements by `step` (not distance), so bullet lifetime/count is unchanged by speed — only range grows. The hit test sweeps the segment from the bullet's previous position (`b.px/py/pz`, updated in the movement loop) to its current one; **on the spawn frame the segment starts at the player's eye**, because the muzzle sits ~13 units ahead of the camera — without the eye→muzzle segment a zombie standing inside claw range is unshootable (the bullet is born behind it). **Headshot = instant kill:** each zombie is tested head-first against a sphere at `mesh.y + HEAD_Y` (11.5) radius `HEAD_R` (2.4), then the chest sphere (`mesh.y + 6`, r 6); a head hit sets `hp = 0`. Every registered hit also calls `spawnBlood` at the impact point (closest point on the bullet segment to the hit sphere's centre). `gunRecoil` / `muzzleFlash.intensity` stay visual-only decays; the pistol slide kicks with recoil.
- **Accuracy / recoil model:** each shot picks a random direction inside a cone in *camera space* (`_sRight`/`_sUp` from `camera.matrixWorld` columns — never world-axis offsets): radius `(SPREAD_BASE 0.035 + gunHeat·SPREAD_BLOOM 0.05) × acc`, where `gunHeat` (0..1) rises +0.22 per shot and cools at 1.4/s (sustained fire blooms, bursts stay tight) and `acc = (ADS? 0.75) × (crouch? 0.75)` — 25% tighter each, stackable. Moving multiplies the spread (not the camera kick) by `movePen`: ×1.25 walking, ×1.5 sprinting. Each shot also kicks the *camera* (rifle 0.012 / pistol 0.016 rad up + small random yaw, scaled by `acc`) via `_kickEuler`, a YXZ euler round-trip identical to the mouse-look handler, so it composes safely with look input (the old "keep recoil off the camera quaternion" rule applies to the *visual* `gunRecoil` only).
- **Movement model:** WASD is combined into one input vector `(fwd, side)` and the displacement is normalized against the *raw* input length, so diagonals are never faster than a single key. Directional weights scale the components before applying: backpedal ×0.7 (−30%), strafe ×0.8 (−20%), forward ×1 — diagonals interpolate between them. Final speed = `player.speed × (sprint 1.7, disabled while crouched/exhausted) × (crouch 0.5 | ADS 0.7) × step`.
- **Stamina:** `STAMINA_MAX` 100. Drains: **sprint 10/s** (only while *actually moving* with Shift — `sprintingNow`, the single source of truth for "effectively sprinting", also drives footstep cadence, gun-bob speed, and the ×1.5 moving-spread penalty), **ADS 3/s**, **melee 15 per swing** (F is ignored below 15). Regen **+12/s** whenever not sprinting and not aiming. Hitting 0 sets `staExhausted`: sprint and ADS lock (ADS auto-lowers, right-click ON is ignored) until stamina regens past `STA_RECOVER` 20 — hysteresis so sprint doesn't flicker around zero. `resetGame` refills to 100.
- **HUD layout:** bottom-left `#barsHud` holds the **HEALTH** bar (`#healthFill`, hp/`PLAYER_MAX_HP`, updated in `updateUI`) and **STAMINA** bar (`#staminaFill`, updated every frame in `update()`); both share `.barShell` styling; the health fill is always red (`#e74c3c`, CSS only — `updateUI` writes width, never color), stamina keeps the green>orange>red scheme (red = exhausted). Bottom-right `#statsHud` (stacked above the radar at `bottom:190px`) holds Score / Ammo / Grenades; `#waveText` stays top-centre; `#pickupFeed` is mid-left. There is no HP text anymore — `hpText` was removed.
- **Crouch (C = toggle, Left Ctrl = hold):** effective state is `crouched = isCrouching || crouchHold` (computed per frame in `update()`; `crouchHold` set/cleared on ControlLeft keydown/keyup — keyup clearing is *ungated* so it can't stick). Crouching drives smoothed `crouchT` which lowers the eye by `CROUCH_DROP` 3.6 (see coordinates bullet), gives the 0.75 accuracy factor, halves move speed (crouch's 0.5 *replaces* ADS's 0.7 — they don't stack to 0.35 — and disables the sprint bonus), and slows/quiets footsteps. Both crouch states are cleared by `releaseInputs()` (pause/blur/reset) like the ADS toggle; `resetGame` also zeroes `crouchT`/`eyeHCur`/`gunHeat`. Ctrl+W (hold-crouch then W → close tab) used to leak to the browser; it's now blocked by fullscreen + Keyboard Lock — see the **Browser-shortcut trap** bullet. The C toggle remains as a no-modifier alternative (and the sole crouch on Firefox/Safari, which lack Keyboard Lock).
- **Input safety:** pointer lock is requested via `requestLock()` (swallows Chrome's NotAllowedError cooldown rejection); `releaseInputs()` clears held keys/mouse/aim on unlock, window blur, and reset. `R` and `G` (grenade) are gated by `!isPaused && !isGameOver`.
- **Browser-shortcut trap (Ctrl+W etc.):** `requestLock()` also calls `enterImmersive()` — **fullscreen** (`documentElement.requestFullscreen`) + **Keyboard Lock API** (`navigator.keyboard.lock(LOCK_KEYS)`). `Ctrl+W`/`Ctrl+R`/`Ctrl+T`… are browser/OS-level shortcuts that `preventDefault()` **cannot** stop; Keyboard Lock is the only reliable interceptor and it only captures system keys **while fullscreen** (Chromium only). `LOCK_KEYS` is the gameplay set (WASD + R/F/G/Q/C/P + T/N + 1/2) **deliberately excluding `Escape`**, so Esc still exits pointer lock (pause) normally. The specific bug this fixes: hold-crouch on Left Ctrl then press `W` was firing `Ctrl+W` and closing the tab. Fallback for Firefox/Safari (no Keyboard Lock): a `beforeunload` guard (`GAME_MODE && !isGameOver && !isPaused`) prompts a confirm before an accidental close/reload during active play. `enterImmersive()` is only ever called from a user gesture (blocker click / Space-restart), as fullscreen requires.
- **ADS (right-click = toggle):** `isAiming` drives a smoothed `aimT` (0..1) in `updateDecor`: FOV lerps `BASE_FOV → BASE_FOV/AIM_ZOOM` (70 → ~58, 20% zoom, `updateProjectionMatrix` on change), the gun lerps from hip (3, −2.5, −6) to centered (0, −0.78, −5.2) so the rear-sight notch + front post line up with screen centre, bob is damped, and the crosshair fades out. Bullets/grenades still derive from the camera's centre ray, but ADS is no longer purely visual: it applies the 0.75 spread/kick factor and the 0.7 move-speed factor (see accuracy bullet). `releaseInputs()` clears the toggle on pause/blur/reset.
- **Hands + reload rigs:** CS-style first-person arms are children of each weapon group (so they inherit bob/ADS/recoil), built from tactical-glove boxes (palm + individual fingers + thumb + trigger finger) and sleeve cuffs/forearms. `placeLimb(mesh, ax..bz)` stretches a unit-z box between two points (midpoint + scale.z + quaternion) — right forearms are placed once (wrist → off-screen bottom-right elbow); **left forearms are re-placed every frame** in `updateDecor` between a fixed elbow anchor and the moving left hand, so arms stay connected through animations. Each weapon has its own reload keyframe track played over **real time** `Date.now() − reloadStartTime` (in sync with its reload `setTimeout` even across pause): rifle `RELOAD_KF` (3.0 s — `leftHand` from `LEFT_HAND_REST`: grab mag → drop below screen → new mag up, `gunMagMesh` rides the hand t 0.35–2.05 → seat → pull `boltHandle` → return) and pistol `PISTOL_KF` (2.2 s — `pLeftHand` from `P_LEFT_REST`: mag out of the grip, `pistolMagMesh` rides t 0.28–1.5 → insert → rack `pistolSlide` → return). Weapon tilt during reload writes `gunRotX`/`gunRotZ`, which are composed with switch-lower and melee rotations at the end of `updateDecor` — never write `grp.rotation` directly elsewhere; `resetGame` zeroes both groups' rotations.
- **Spawning:** `dt` accumulator in `update()` fires `spawnZombie` every `wave.spawnInterval`s; capped at `wave.maxZombies`. Zombies spawn **outside the fence** (mostly on the side nearest the player) and enter via a two-phase state machine: `state:'jumping'` arcs them over the fence (sine parabola, lerping `s*→l*`), then `state:'chasing'` runs the normal ground AI. The jump arc is generalized — `y = baseY + lerp(jumpY0, jumpY1, t) + sin(πt)·arcH` — and is **reused as a fountain vault**: a chasing zombie blocked by the basin wall (`resolveObstacles` returns true) re-enters `'jumping'` onto the basin top when `segmentHitsFountain` confirms the path to the player crosses it (`vaultCd` 1.5s prevents ping-pong loops). `z.groundY` tracks the current floor (0 or `FOUNTAIN.topY`) and snaps to 0 when they walk off the rim. **Zombies attack by clawing, not kamikaze:** a chasing zombie stops advancing inside claw reach (`player.radius + 3.0`) and, when `z.attackCd` ≤ 0 and within `player.radius + 3.5`, deals 1 damage, sets `attackCd = CLAW_CD` (2 s per zombie), and plays a one-arm swipe (`clawT`/`clawSide`, animated in `animateZombieRig` over the current pose). While standing in claw range (`z.moving === false`) the walk cycle stops entirely — legs/bob damp to rest and only a small breathing sway remains. The zombie persists and keeps clawing. Jumping/vaulting zombies are shootable but can't claw until `chasing`. The bullet hit-test offsets both spheres from `mesh.y` (chest `+ (isModel ? 6 : 0)`, head `+ HEAD_Y`) so it works mid-air, on grounded models (feet at y=0), and on the basin top.
- **SFX:** always play via `playSFX(sfx, vol = 0.7)`, which round-robins a small per-clip pool (max 8 cloned `Audio` nodes) so overlapping sounds work without per-shot allocation. Clips in `assets/sounds/`: `gun-shoot`, `pistol-shoot`, `grenade-explode`, `reload` (shared by both weapons), `pick-up-item` (drop pickup), `smash-melee-attack` (melee swing, plays even on a whiff), `throwing-grenade`, `player-footstep` (walk/sprint cadence via `footT` timer + landing after a jump, at reduced volume), `zombie-attack-melee` (claw hit — plays together with `jokowi-kaget`, the player-damage scream `sfxHit`), `grenade-rolling` (once per grenade at first floor contact, only within 90 units — see grenade bullet), `zombie-step` (**one global source, not per-zombie**: every ~0.55 s the `zStepT` timer plays a single step for the nearest moving chasing zombie within 120 units, volume scaled by distance — keep it this way for performance). `zombie-spawn.mp3` exists on disk but is **no longer played**.
- **Zombie model = procedural blocky human** (story: civilians turned by aliens). `buildHumanZombie()` assembles one per spawn from shared `ZG` geometries (~13 units tall, feet at y=0, facing +Z so the AI's `lookAt` works directly): torso, multi-material head (shared canvas face texture `zombieFaceTex` on the +Z face, index 4, tinted by skin tone), hip/knee/shoulder pivot groups, and profession accessories. Six `ZOMBIE_VARIANTS`: warga (random shirt colors), pekerja (orange vest + stripe + yellow helmet), polisi (cap + brim + gold badge), pedagang (apron + caping cone), penjaga toko (blue vest + name tag), kantoran (tie). Skin from `ZOMBIE_SKIN_TONES`; every material is per-instance (`offsetHSL` grime jitter); `disposeZombie` handles both single and **array** materials (the head) — never dispose `ZG` geometries or `zombieFaceTex`. **Zombies show no damage feedback by design** (no hit tint, no low-hp color — the old `tintZombie` was removed on purpose; the player must not be able to tell how hurt a zombie is). `isModel` stays `true` on every zombie so hit constants (chest +6, hitR 6, head +11.5 r 2.4, baseY 0) are untouched.
- **Zombie animation:** `animateZombieRig` rotates the pivot groups' `rotation.x` directly (lateral axis = local X because the rig faces +Z and the group is `lookAt`-ed at the player) — no quaternion math, no `AnimationMixer`. Walk = alternating thigh/shin/arm swing + body bob (`inner.position.y`) + head sway; base arm pose is the classic zombie forward reach (−1.15 rad); jump/vault = static tuck. Amplitudes are the `SW`/`KN`/`AR` constants.
- **`resetGame()`** must clear every entity array and `scene.remove` its meshes — keep it in sync when adding entity types.

## Editing guidance (from AGENTS.md)

- Keep changes compatible with a single static page. Do not introduce build tools, frameworks, or package dependencies unless the user explicitly asks to modernize the project structure.
- Only split inline script/CSS into separate files if the user requests it.
- Preserve existing controls (WASD move, Shift sprint, mouse look, left-click fire, right-click toggle aim-down-sights, 1/2/Q switch weapon, F melee, G grenade, R reload, C toggle crouch / Left-Ctrl hold crouch, Space to jump — or restart on the game-over screen) and core mechanics unless the user asks for gameplay changes.
